## Why this release

v0.13.1 shipped the engine-events outbox in the `public` schema with implicit prefixed table names
(`engine_events`, `engine_lock`, `workflow_workflows`, …). This works but conflates module
ownership: every module's tables sit in one namespace, identified only by naming convention.

For v0.14.0 we add an `auth` module with ~10 new tables. Adding them to `public` alongside
workflow's tables would make the picture worse: more prefixed names, no enforced module boundary, no
per-module backup/permission story, no clean "this module is disabled" lifecycle.

This release establishes the storage isolation model that auth (and any future module) lands into:

- **PostgreSQL**: one database, one schema per module (`engine`, `workflow`, `auth`). Tables
  addressed schema-qualified (`engine.events`, `auth.users`). Cross-schema transactions are atomic —
  preserves the v0.13.1 publish-on-commit guarantee.
- **SQLite**: one file per module (`engine.db`, `workflow.db`, `auth.db`), attached to one
  connection on startup. Cross-attached-DB transactions are atomic under SQLite's default journal —
  same guarantee.
- **Trait abstraction**: module SQL emits identical schema-qualified syntax (`auth.users` reads
  identically on both backends — PG sees a schema, SQLite sees an attached database, syntax
  matches).

## Architectural decisions (not re-litigated)

These were settled in conversation with the user before this plan was written:

1. **Schemas/attach over per-module databases.** PG cross-schema transactions (and SQLite
   cross-attached-DB transactions) preserve the v0.13.1 atomic publish-on-commit guarantee that
   separate databases would lose. Three separate PG databases would force eventual-consistency for
   cross-DB events; schemas give the same isolation feel without that cost.
2. **Drop redundant prefixes.** `engine.engine_events` is doubly redundant; the schema namespaces.
   So `engine_events` → `engine.events`, etc.
3. **`engine` schema is engine-core infrastructure, always present.** Lives regardless of which
   functional modules are enabled. Other modules' schemas are created/skipped based on
   `engine.modules` enablement.
4. **Auth does NOT write to `engine.events`.** Auth emits its own real-time events through a thin
   per-module mechanism if/when needed; for compliance, auth uses its own `auth.audit` table.
   Engine.events stays as workflow's notification mechanism (the v0.13.1 design honored).
5. **Active dev, breaking change OK.** No migration guide for v0.13.1 → v0.13.2. Fresh install or
   local data loss is acceptable.
6. **Multi-node PG works today** via `pg_try_advisory_lock(1)` for leader election +
   `FOR UPDATE SKIP LOCKED` for distributed task claiming + PG `LISTEN`/`NOTIFY` for cross-instance
   event propagation. v0.13.2 adds `engine.instances` as a _visibility_ table only, not new
   coordination capability.

## Boot sequence (revised)

```
1. Open engine storage
   - PG: connect to configured URL; CREATE SCHEMA IF NOT EXISTS engine
   - SQLite: open engine.db file (path from config; default ./engine.db)

2. Apply engine schema migrations
   - Bootstrap engine.migrations table itself
   - Run all engine.* DDL up to current version

3. Read enabled modules
   - SELECT name, version, config FROM engine.modules WHERE enabled = TRUE

4. For each enabled module:
   - PG: CREATE SCHEMA IF NOT EXISTS <module>
   - SQLite: ATTACH DATABASE '<module>.db' AS <module> (path = next to engine.db)
   - Run module migrations (tracked in engine.migrations under module name)

5. Wire trait routing
   - WorkflowStore impl bound to workflow schema/attach
   - AuthStore impl bound to auth schema/attach (when auth lands in v0.14.0)

6. Engine-level multi-node coordination
   - PG: pg_try_advisory_lock(1) for leader election (existing v0.13.1 path)
   - SQLite: engine.lock single-row exclusive (existing v0.13.1 path)
   - Insert into engine.instances on startup, refresh on timer, DELETE on
     graceful shutdown

7. Mount HTTP routers from each enabled module
8. Start scheduler, workers, etc.
```

## Risks & mitigations

| Risk                                                                      | Likelihood                                                                                                                    | Mitigation                                                                                                                                                  |
| ------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ALTER TABLE … SET SCHEMA` drops a constraint / index in PG               | Low                                                                                                                           | PG documentation states constraints follow; verify in test                                                                                                  |
| SQLite ATTACH limit (default 10 attached DBs)                             | Very low at present (3 modules max planned)                                                                                   | Document the limit; bump `SQLITE_LIMIT_ATTACHED` if we ever exceed                                                                                          |
| SQLite ATTACH atomicity claim                                             | Confirmed in docs but verify in test — write to two attached DBs in one transaction, kill mid-commit, verify both rolled back | Phase 5 test case                                                                                                                                           |
| LISTEN channel name change after `engine_events` → `engine.events` rename | NOTIFY channel name is configured in code (`assay_engine_events_<ns>`), not derived from table name                           | No change needed                                                                                                                                            |
| Hidden hardcoded table name in seed script or test fixture                | Medium                                                                                                                        | grep for `engine_events`, `workflow_workers`, etc. in `crates/**` and `tests/**`; fix all                                                                   |
| Moving `api_keys` to wrong schema (engine vs workflow)                    | Medium                                                                                                                        | Phase 2 task: read existing usage; if used by workflow REST API auth, move to `workflow.api_keys`; if used for engine admin only, move to `engine.api_keys` |

## Out of scope (deferred)

- **Per-module backend override** (auth on cluster A, workflow on cluster B). Drops atomic
  publish-on-commit; the user explicitly de-prioritised this. Always solvable later by running
  multiple engine instances.
- **MySQL backend.** Out of plan; would require non-trivial work on the notification path (no native
  NOTIFY/LISTEN equivalent).
- **Engine config in DB.** Toml at startup is fine; runtime-mutable config is a later concern.
- **Dashboard cluster view.** `engine.instances` table exists; UI consumption lands in v0.14.0 phase
  8 alongside auth dashboard panes.
- **Auth module content.** v0.14.0 (plan 12 phases 4–8).

# Plan 15 ‚Äî v0.2.0 route + auth unification

**Status**: Approved, dispatching in 5 slices **Branch**: `feature/v0.14.0-auth` (continuing the
umbrella v0.2.0 PR ‚Äî no new branch) **Estimated effort**: ~10‚Äì13h agent time across 5 commits

## Why this plan exists

The v0.2.0 work-in-progress shipped functional auth + workflow + engine surfaces but with three
architectural problems that emerged once the user reviewed the URL surface:

1. **Doubled URL segments** ‚Äî `/auth/admin/auth/users` because the admin router defines
   `/admin/auth/...` then it's mounted under `/auth`. Same problem for `/auth/admin/auth/sessions`,
   `/auth/admin/auth/biscuit`, `/auth/admin/auth/jwks`, `/auth/admin/auth/zanzibar/*`,
   `/auth/admin/auth/audit`. Real bug, not just cosmetics.
2. **Workflow API at top of `/api/v1/*`** ‚Äî `/api/v1/workflows`, `/api/v1/namespaces`,
   `/api/v1/schedules`, etc. Engine API is at `/api/v1/engine/*` (correct namespacing). Workflow
   should mirror.
3. **Three independent auth systems** ‚Äî `admin_api_keys` (engine.toml static), `workflow.api_keys`
   (DB-backed), `auth` module (sessions/JWT/passkeys/Zanzibar). The `auth` module should be THE
   identity layer for everything: operator login, workflow API access, OIDC IdP for downstream apps.
   Per-endpoint authorization moves to Zanzibar role checks.

Per-user direction: no back-compat shims, no kept-for-compat URL aliases ‚Äî clean refactor.

## Final URL layout (locked)

```
EXTERNAL-FACING (OIDC spec ‚Äî kept where they are; published in discovery doc)
  /auth/.well-known/openid-configuration
  /auth/.well-known/jwks.json
  /auth/{authorize, authorize/consent, token, userinfo, revoke, introspect, logout}
  /auth/oidc/upstream/{slug}/{start, callback}        ‚Üê federation, spec-y enough to keep here

INTERNAL ENGINE API (everything callable by Lua client + dashboards)
  /api/v1/engine/core/{info, modules, instances, audit, config}
  /api/v1/engine/core/active-modules                  ‚Üê rename from /api/v1/modules
  /api/v1/engine/core/health                          ‚Üê rename /healthz; keep /healthz as 1-line redirect for k8s probes

  /api/v1/engine/workflow/{namespaces, workflows, workflows/{id}/...}
  /api/v1/engine/workflow/{activities, workers, tasks, schedules, queues, api-keys}
  /api/v1/engine/workflow/{health, version, openapi.json, docs}

  /api/v1/engine/auth/{login, logout, whoami}
  /api/v1/engine/auth/passkey/{register, auth}/{start, finish}
  /api/v1/engine/auth/admin/{users, sessions, biscuit, jwks, zanzibar/*, audit}
  /api/v1/engine/auth/admin/oidc/{clients, upstream}/...
```

## Auth model (locked)

ONE `auth` module. THREE audience patterns:

- **Operator** ‚Äî logs in via `/api/v1/engine/auth/login` (or federated upstream IdP), session cookie
  set, gates engine admin via Zanzibar (`engine#admin` role)
- **Workflow user** ‚Äî same login, gated by Zanzibar (`workflow:<ns>#user|admin`)
- **Consumer app** ‚Äî OIDC code flow at `/auth/authorize`, gets JWT, validates against
  `/auth/.well-known/jwks.json`

Multi-IdP support: `auth.upstream_providers` table holds N entries (Google, Microsoft, customer's
existing Ory Hydra, anything OIDC). Login page renders one button per registered upstream.

`admin_api_keys` (engine.toml) becomes break-glass / bootstrap only. `workflow.api_keys` table
retired entirely (replaced by JWT/session check + Zanzibar).

## Rollback strategy

- Each slice is its own commit. `git revert <sha>` per slice if a slice needs unwinding.
- Slice 3 is the breaking-change cliff (engine refuses to start without admin). Operators on the
  umbrella PR who already deployed will need to either run `bootstrap-admin` or set `admin_api_keys`
  in toml. Documented in CHANGELOG.

## CI consequences

- Slice 1 push: CI re-runs full suite. If previously-failing tests pass after URL fixes, that's
  expected (URL fixtures were stale).
- Slice 2 push: tests for the new auth gate land. Previous tests that hit admin endpoints with
  api-keys still work (fallback path).
- Slice 3 push: any test that boots engine without admin_api_keys + no users will fail. All such
  tests need a `bootstrap-admin` call in their fixtures (handled in Slice 3 itself).
- Slice 4 + 5: new Lua tests run in their own CI job; existing Rust+Playwright suites unaffected.

## Out of scope (deferred to v0.2.1+)

- Federated logout (RFC 8414 single-sign-out) across upstream IdPs ‚Äî login federation lands here,
  full logout federation is a polish item
- Claim ‚Üí role mapping from upstream JWTs (e.g. Google's `groups` claim ‚Üí engine Zanzibar tuples) ‚Äî
  admin can manually assign roles after first login for now
- Live module enable/disable without engine restart ‚Äî the `toggle` endpoint marks the row, restart
  still required

# Plan 08: Dynamic workflow definitions via API (v0.12)

## Problem

Today a workflow type only exists after someone:

1. Writes Lua source in a `worker.lua` file.
2. Bakes that file into a worker container image.
3. Deploys the image.
4. The worker calls `workflow.define("MyWorkflow", fn)` at boot and
   becomes able to execute that type.

This locks workflow authoring to anyone with the ability to build and
deploy worker images. For two valuable emerging use cases, that gate is
the wrong shape:

- **AI agents.** An agent that can generate Lua can't usefully ship it
  today — it would have to build an image, push to a registry, and
  trigger a rollout. The round-trip is minutes-to-hours when the agent
  wants "POST and run".

- **Operator-authored ad-hoc automations.** Someone writing a one-off
  migration script or a quick SRE workflow doesn't need the same
  image-build discipline as long-lived production workflows. A
  code-editor-in-the-dashboard flow is much closer to the task at hand.

## Goal

Let workflow definitions be **created, versioned, and executed entirely
via the engine's HTTP API**, with no worker rebuild required. Preserve
every current guarantee (deterministic replay, crash safety,
namespace-scoped execution) and add no new Lua surface — dynamic
definitions use the exact same `workflow.define` / `ctx:*` API as
file-based ones.

## Non-goals

- A visual node-graph editor. v0.12 targets Lua-authored workflows
  with a text editor in the dashboard. Node-graph editing would be a
  separate product that generates Lua and POSTs it via the same API.
- Removing file-based workflows. They stay. Production workflows that
  live in a repo and ship via image rebuilds are still the right shape
  for mission-critical cases; dynamic definitions complement them, not
  replace them.
- Cross-language SDKs. Still Lua-only.

## Architecture

```
  Operator / AI agent
         │
         │ POST /api/v1/workflow-definitions
         │  { name, lua_source, version? }
         ▼
  ┌───────────────────────────────────────────────┐
  │ Engine (same binary as today)                │
  │                                               │
  │  ┌────────────────────┐   ┌────────────────┐ │
  │  │ Postgres store      │   │ API            │ │
  │  │ - workflows         │◀──│ POST/GET/PUT   │ │
  │  │ - events            │   │ /workflow-     │ │
  │  │ - workflow_defs     │◀──│  definitions   │ │
  │  │   (NEW table)       │   │                │ │
  │  └────────────────────┘   └────────────────┘ │
  │                                               │
  │  Dispatcher (unchanged): hands tasks to       │
  │  workers registered for the target namespace. │
  └───────────────────────────────────────────────┘
         │
         │ Task dispatch (same protocol as today)
         ▼
  ┌───────────────────────────────────────────────┐
  │ Generic Worker Pool (Option A, see below)     │
  │                                               │
  │  - Registered for namespace X, any type       │
  │  - Poll /workflow-tasks (as today)             │
  │  - On claim: GET /workflow-definitions/{name} │
  │              /{version} → Lua source          │
  │              (cached in-process per version)   │
  │  - mlua compiles + executes the definition    │
  │  - Deterministic replay: same (name, version) │
  │    always yields the same bytecode            │
  └───────────────────────────────────────────────┘
```

### The worker sourcing decision — Option A

The central v0.12 question is where the worker that runs a
dynamically-created definition comes from. Three options were
considered:

| Model                           | Description                                                                                                                             | Operator burden                       | AI-agent fit    |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- | --------------- |
| **A. Generic per-namespace pool** | A single generic worker deployment per namespace. Workers register as "willing to execute any type in this namespace." On claim, worker fetches the definition source from the engine and compiles in-process. | Deploy worker pool once per namespace. | ✅ Ideal         |
| **B. Per-type workers (status quo)** | Each workflow type needs a pre-configured worker. Dynamic definitions still require image builds.                                   | Every new definition = rebuild.      | ❌ No            |
| **C. Engine-embedded workers**   | Engine has an internal worker pool; no separate process. Simpler ops but blurs engine/worker roles and conflicts with scaling model.     | Zero.                                | ✅ But awkward   |

**Picked: A.** Keeps the engine/worker split clean, supports AI agents
out of the box, and the resource-isolation and sandbox concerns are
tractable with existing primitives (mlua isolation, per-task timeouts,
Postgres-level quotas).

A generic worker:

- Registers on startup with `workflow.listen({ queue = "...", namespace = "...", accepts = "*" })`.
- On claiming a workflow task, extracts the definition name+version from
  the task payload (dispatcher supplies it).
- If not yet in the per-process compile cache, `GET /api/v1/workflow-definitions/{name}/{version}`, compile via mlua, cache.
- Execute the handler normally. Deterministic replay works because
  `(name, version)` is pinned on the workflow record at start time.

### Determinism contract

Every workflow run pins a **specific definition version** at start.
That version is stored on the `workflow` record and re-fetched on every
replay. Updating the definition creates a new version row; in-flight
runs continue executing their pinned version.

```
workflow_definitions
──────────────────────
  id           serial PK
  namespace    text           -- scoped, same as workflows
  name         text
  version      integer        -- monotonic per (namespace, name)
  lua_source   text
  created_at   timestamptz
  created_by   text           -- from the auth context
  is_active    boolean        -- only one version per (namespace, name) is "active"
                              -- new workflow.start() picks the active version

  PRIMARY KEY (id)
  UNIQUE       (namespace, name, version)
```

On `workflow.start({ workflow_type })`:

1. Engine resolves `(namespace, workflow_type)` to the active
   `workflow_definitions` row.
2. Stores `definition_version` on the workflow record.
3. Workers fetch `/api/v1/workflow-definitions/{name}/{version}` on
   first claim, cache in-process.

On definition update:

1. `PUT /workflow-definitions/{name}` creates a new version row.
2. The new version becomes active.
3. Already-running workflows continue to fetch their pinned version —
   replay determinism preserved.

### Visuals for authoring

v0.12 MVP is an **in-browser code editor** in the dashboard. A new
"Definitions" sidebar nav. New definition = write Lua in an editor
pane, click Save, POST to the API. List of definitions shows (name,
active version, last edited, created by). Click a definition → full
source + version history + "start a run from this type" button.

**Editor choice: Monaco vs CodeMirror.**

| Feature                           | Monaco                                   | CodeMirror 6                               |
| --------------------------------- | ---------------------------------------- | ------------------------------------------ |
| Bundle size (minified + gzipped)  | ~3 MB (ouch)                             | ~150 KB (mix-and-match modules)            |
| Lua syntax highlighting           | Built in                                 | via `@codemirror/legacy-modes`             |
| Autocomplete / IntelliSense API   | Excellent (`registerCompletionProvider`) | Good (`autocomplete` extension)            |
| LSP-style hover docs              | Excellent                                | Possible via `hoverTooltip`                |
| AI-assistance integration         | Has a built-in "inline completions" API  | Standard extensions work; a bit more glue  |
| Theming                           | Good, JSON-configurable                  | Excellent, CSS-variable-driven             |
| Load on every page                | No — lazy-loaded per editor              | Same — lazy-loaded                         |

**Picking CodeMirror 6.** The 3 MB vs 150 KB difference matters for a
dashboard that's already lazy-loaded on demand, and CodeMirror's
mix-and-match module design keeps us from shipping features we're not
actually using. For Lua specifically, neither has deep built-in
intelligence; we'd be building the completion/hover providers either
way.

**AI assistance — three layers:**

1. **Stdlib-aware autocomplete.** The editor ships with completion
   for assay's stdlib surface — `workflow.define(…)`, `ctx:*`
   methods, every `assay.*` module's API. Generated from the same
   metadata `assay modules` + `assay context` CLI use, so the list
   stays in sync with the runtime without hand-curation.

2. **Inline "generate from prompt" panel.** A `⌘K` shortcut opens a
   prompt input that accepts natural language ("a workflow that waits
   for approval then calls the ECR retag activity") and streams Lua
   suggestions back. The dashboard is the client; the actual LLM call
   is deployment-configurable — operators supply an endpoint + API key
   via new env vars (`ASSAY_AI_COMPLETION_URL`, `ASSAY_AI_COMPLETION_TOKEN`)
   and the dashboard posts through that. No hardcoded provider.

3. **Validation + lint inline.** On each save attempt the engine's
   `POST /workflow-definitions?dry_run=true` compiles the Lua via
   mlua and returns parse errors with line numbers. The editor
   underlines errors immediately — same feedback loop a local
   developer gets, without requiring a local toolchain.

The AI-assistance layer is **optional** — dashboards deployed without
`ASSAY_AI_COMPLETION_URL` simply don't show the `⌘K` panel. The
stdlib autocomplete and dry-run validation ship regardless; they're
static knowledge about assay itself and don't need a cloud LLM.

No drag-and-drop, no node graph. The code is the truth; the editor is
ergonomic surface around the API. A node-graph product, if ever built,
lives on top of the same REST API and generates Lua that the editor
can round-trip.

## API

```
POST   /api/v1/workflow-definitions
       { namespace, name, lua_source, make_active? = true }
       → 201 { version }

GET    /api/v1/workflow-definitions?namespace=&active=
       → [ { name, version, created_at, created_by, is_active } ]

GET    /api/v1/workflow-definitions/{name}
       ?namespace=
       ?version=    # optional; omitted returns the active version
       → { name, version, lua_source, created_at, created_by, is_active }

PUT    /api/v1/workflow-definitions/{name}
       { namespace, lua_source }
       → 201 { version }

POST   /api/v1/workflow-definitions/{name}/activate
       { namespace, version }
       → 200 (switches the active version)

DELETE /api/v1/workflow-definitions/{name}
       { namespace, version? }  # no version = delete all
       → 204
```

All endpoints require the new auth scope **`workflow.definitions.write`**
(write operations) or the existing viewer scope (reads).

## Security

- **Auth scope `workflow.definitions.write`** — separate from the
  existing trigger / approve scopes. Operators grant this only to
  identities they trust to author code.
- **Per-namespace quotas** — max definitions per namespace, max source
  size per definition, max total source bytes (Postgres row bloat
  protection), max new-definition rate per identity.
- **Sandbox** — mlua already confines Lua to the stdlib assay provides.
  No new privilege escalation risk beyond what existing file-based
  workflows already have. Dynamic definitions cannot call Rust FFI
  directly; they use the same `assay.*` modules.
- **Audit log** — every create / update / activate recorded with
  `{ identity, timestamp, name, namespace, version }`. Surfaced in the
  dashboard and queryable via `/api/v1/audit/workflow-definitions`.
- **Optional source signing** — future extension: require definitions
  to carry a signature from a trusted identity before workers will
  execute them. Not in MVP; tracked as a follow-up.

## Worker hot-reload

Workers under the generic pool model don't need reload for *new*
definitions (they fetch on first claim). They do need to handle:

- **Activated-version change mid-run.** Existing runs continue at their
  pinned version. New runs pick up the newly-active version on their
  next claim.
- **Cache eviction.** LRU cache keyed by `(namespace, name, version)`.
  Size bounded so a rogue namespace can't starve other namespaces.
- **Stale-definition GC.** If a pinned version is deleted from the
  store mid-run, the worker fails the task with a clear error. The
  workflow goes to FAILED with reason `definition_deleted`.

## Phased rollout

### Phase 1 — Engine: storage + CRUD API (v0.12.0)

- Postgres table + migration.
- REST endpoints (create / read / update / activate / delete).
- Auth scope `workflow.definitions.write` + enforcement.
- Per-namespace quotas (configurable env var defaults).
- Audit log table + endpoint.
- Tests: round-trip CRUD, version pinning, quota enforcement, auth scope.

### Phase 2 — Worker: generic pool mode (v0.12.0)

- `workflow.listen({ accepts = "*" })` mode.
- Per-version compile cache + LRU eviction.
- Integration tests: create definition, start workflow, generic worker
  picks it up, runs to completion.

### Phase 3 — Dashboard: Definitions view + code editor (v0.12.1)

- New sidebar nav entry: Definitions.
- List view + per-definition detail + version history.
- **CodeMirror 6 editor** — lazy-loaded, with Lua syntax highlighting
  via `@codemirror/legacy-modes`. Bundle cost ~150 KB, only on the
  Definitions view.
- **Stdlib-aware autocomplete.** Completion provider sourced from the
  same metadata `assay modules` / `assay context` use, so the list
  stays in sync with the runtime without hand-curation.
- **Inline lint** — on save-attempt, `POST /workflow-definitions?dry_run=true`
  compiles via mlua and returns parse errors. Editor underlines errors
  immediately (same feedback loop a local developer gets).
- **Optional AI-assist panel (⌘K).** Dashboard posts prompt + current
  source through a deployment-configured endpoint:
  - `ASSAY_AI_COMPLETION_URL` — operator-supplied endpoint that accepts
    `{ prompt, context, model? }` and streams Lua.
  - `ASSAY_AI_COMPLETION_TOKEN` — bearer token the dashboard forwards.
  Dashboards without these env vars simply don't show the panel. No
  hardcoded provider.
- "Start a run" button — wraps `workflow.start({ workflow_type })`.

### Phase 4 — Safety rails (v0.12.2)

- Source signing (opt-in per namespace).
- Rate limiting on the authoring API.
- Per-definition execution timeouts and resource caps.
- "Draft vs published" lifecycle (author can iterate without affecting
  active version).

## Estimate

Roughly 2 weeks for Phases 1 + 2 (MVP: engine + generic worker mode).
Another week for Phase 3 (dashboard). Phase 4 is ongoing polish and can
ship piecemeal across v0.12.x patch releases.

## Open questions

- [ ] Do we need sandbox isolation beyond mlua's default
      (e.g. resource limits per coroutine via Lua hooks)? Revisit after
      Phase 2 shows real-world footprint.
- [ ] What happens when a caller POSTs Lua that doesn't compile? Fail
      the create, or store as draft with an error flag? Leaning toward
      "fail fast" for MVP — drafts are Phase 4.
- [ ] Namespace administration: who can create a new namespace to host
      dynamic definitions? Existing `workflow.namespaces.*` API already
      requires admin scope; reuse.
- [ ] Signed definitions: which signature scheme? PGP is heavy,
      Sigstore is cleaner but adds deps. Pick in Phase 4.

# Plan 19: v0.15.1 — `process.spawn_pty` + server-side WebSocket upgrade in `http.serve`

(Folded into v0.15.1 alongside the Linux/sysd/cgroup work from plan 18 — same release, additive
surface.)

## Problem

Operator-facing Lua scripts that want to host an in-browser terminal — wire up `xterm.js` to a
`machinectl shell <name>` PTY through a single static-binary deploy — can't do it with assay 0.15.x
today. Two missing primitives:

1. **No PTY allocation in `process.spawn`.** `spawn` writes stdout to a file path and that's it. No
   bidirectional pipes, no stdin write handle, no PTY semantics. `script -qfec` could fake a PTY in
   a subprocess but loses `TIOCSWINSZ` resize support and many programs detect the indirection.
2. **`http.serve` has no server-side WebSocket upgrade.** The `ws` builtin uses
   `MaybeTlsStream<tokio::net::TcpStream>` (the tungstenite _client_ type), so Lua scripts can
   connect TO a websocket server but not accept incoming connections. `http.serve` already handles
   upgrade-headers for SSE (`{sse = function(send) ... end}`); it should grow a parallel
   `{ws = function(conn) ... end}` for full-duplex.

The real-world consumer that wants both: an "Open Shell" button on a container management dashboard
([fcar/knowhere](https://github.com/fcartech/knowhere) is the first known caller). It needs:

- Browser side: `xterm.js` over a `WebSocket` connection
- Server side: per-connection PTY spawned with `machinectl shell <name>` (or
  `nsenter -t <leader_pid> -m -u -i -n -p -- bash`), bytes piped both directions, terminal resize
  delivered as `TIOCSWINSZ` ioctl

Both pieces are well-understood Rust patterns. Ship them as additive builtins (no breaking changes)
in v0.15.1.

## Goal

Two new Lua surfaces in v0.15.1:

```lua
-- 1. Spawn a child process attached to a fresh PTY, return a duplex handle.
local pty = process.spawn_pty({
  cmd  = "machinectl",
  args = { "shell", "agentx" },
  cwd  = "/",                        -- optional
  env  = { TERM = "xterm-256color" },-- optional
  cols = 80,
  rows = 24,
})

pty:write(bytes)         -- write to PTY stdin
local chunk = pty:read(opts?)        -- read from PTY stdout (blocking;
                                     -- async-aware so it yields the Lua
                                     -- coroutine, doesn't block the runtime)
                                     -- opts = {timeout_ms?}; returns nil on EOF
pty:resize(cols, rows)   -- TIOCSWINSZ
pty:close()              -- SIGHUP the child + close fds
pty.pid                  -- read-only pid for status/audit
```

```lua
-- 2. Server-side WebSocket upgrade returned from an http.serve handler:
http.serve(8080, {
  GET = {
    ["/api/machines/*/shell"] = function(req)
      return {
        ws = function(conn)
          -- conn:read() returns next message (text or binary), nil on close
          -- conn:write(bytes, opts?) sends a message; opts = {binary?=true}
          -- conn:close() closes the connection cleanly
          local pty = process.spawn_pty({...})
          while true do
            local msg = conn:read()
            if not msg then break end
            pty:write(msg)
            local out = pty:read({timeout_ms = 50})
            if out then conn:write(out, {binary = true}) end
          end
          pty:close()
        end,
        -- Optional headers for the upgrade response
        headers = { ["X-Shell-Allowed"] = "true" },
      }
    end,
  },
})
```

A typical "Open Shell" handler is ~30 lines of Lua on top.

## Non-goals

- **Hard PTY-resize semantics from the Lua side** beyond `pty:resize(cols, rows)`. No
  ioctl-passthrough surface; just rows+cols.
- **Pseudoterminal pair API** (`openpty()` + master/slave fds exposed). The `spawn_pty` table-handle
  interface is sufficient for every dashboard / shell use case; a raw fd surface would expand the
  unsafe area substantially.
- **WebSocket subprotocol negotiation, compression, ping/pong customisation.** Default behaviour
  only: tungstenite handles ping/pong automatically; we expose text/binary frames and clean close.
  Custom subprotocols and per-message deflate land later if needed.
- **Cross-platform PTY.** Linux + macOS only via `nix::pty::openpty`. Windows needs `ConPTY` which
  is a different API surface — defer.
- **TLS termination on the WebSocket upgrade.** `http.serve` is plaintext today; put a reverse proxy
  (Caddy, nginx, traefik) in front for `wss://`. Native TLS is a separate, larger plan.

## Design

### Module 1 — `process.spawn_pty` (Rust builtin extension)

Goes into `crates/assay/src/lua/builtins/process.rs` as a new function alongside the existing
`spawn`. Backed by:

- `nix = "0.30"` (already a transitive dep through `procfs`; verify & pin explicitly if needed) for
  `nix::pty::openpty`, `nix::sys::ioctl`, `nix::sys::signal::Signal`
- `tokio = "1"` (already a dep) for async I/O over the master fd
- `mlua` `UserData` for the handle

#### Lua surface

```lua
local pty = process.spawn_pty(opts)  -- returns a PtyHandle UserData
```

`opts` table:

| Field  | Type                 | Required | Default   | Notes                      |
| ------ | -------------------- | -------- | --------- | -------------------------- |
| `cmd`  | string               | yes      | —         | argv[0]                    |
| `args` | string[]             | no       | `[]`      | argv[1..]                  |
| `cwd`  | string               | no       | inherited | working directory          |
| `env`  | table<string,string> | no       | inherited | child env vars (key/value) |
| `cols` | integer              | no       | 80        | initial PTY columns        |
| `rows` | integer              | no       | 24        | initial PTY rows           |

#### Handle methods

| Method                                | Behavior                                                                                                                                                          |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pty:write(data: string)`             | Async write to the PTY master fd. Returns nil on success, errors on broken pipe.                                                                                  |
| `pty:read(opts?)` → `string \| nil`   | Async read from the PTY master fd. Returns next chunk (up to 4 KiB), or nil on EOF / child exit. `opts.timeout_ms` (default 0 = block forever; coroutine-yields). |
| `pty:resize(cols, rows)`              | Issues `TIOCSWINSZ` on the master fd. Errors if cols/rows < 1.                                                                                                    |
| `pty:close()`                         | Sends SIGHUP to the child, closes both fds. Idempotent.                                                                                                           |
| `pty:wait()` → `(exit_code, signal?)` | Async wait for child to exit. Coroutine-yields. After this returns, `pty:close()` is a no-op.                                                                     |
| `pty:is_alive()` → bool               | Non-blocking check via `kill(pid, 0)`                                                                                                                             |
| `pty.pid`                             | Read-only integer (mlua UserData metatable index)                                                                                                                 |

#### Implementation notes

- **PTY allocation**: `nix::pty::openpty(None, None)` returns `(master_fd, slave_fd)`. Set master to
  non-blocking via `fcntl(O_NONBLOCK)`.
- **Child setup**: standard fork+exec, but in the child:
  - `setsid()` to create a new session
  - `dup2(slave_fd, 0/1/2)` to wire stdio
  - `ioctl(slave_fd, TIOCSCTTY, 0)` to make it the controlling terminal
  - `close(master_fd)`, `close(slave_fd)` after dup
  - `execvp(cmd, args)`
- **Async I/O**: wrap master_fd in `tokio::io::unix::AsyncFd` (manages epoll registration). `read`
  and `write` go through the AsyncFd's readable/writable futures.
- **Resize**:
  `nix::libc::ioctl(master_fd, TIOCSWINSZ, &winsize { ws_row, ws_col, ws_xpixel, ws_ypixel })`. Send
  `SIGWINCH` to the child afterward (kernel doesn't always do this — be explicit).
- **Lifetime**: `PtyHandle` owns the master fd + child pid. `Drop` calls `pty:close()` semantics so
  a Lua-side forgotten handle still cleans up.
- **Linux + macOS only**: `#[cfg(any(target_os = "linux", target_os = "macos"))]` on the impl
  module. Other targets register the `spawn_pty` function returning
  `mlua::Error::runtime("process.spawn_pty: Linux/macOS only")`.

#### Tests

```rust
#[cfg(test)]
mod tests {
    // 1. Spawn `cat`, write "hello\n", read it back
    // 2. Spawn `echo wrap`, no input, read returns "wrap\r\n" then nil on EOF
    // 3. Spawn `bash -c 'tput cols && tput lines'` with cols=120 rows=40,
    //    parse output, assert 120/40 (verifies winsize was applied)
    // 4. Spawn `bash -c 'sleep 30'`, call :close(), :wait() returns SIGHUP
    // 5. spawn_pty -> drop without :close() -> child gets SIGHUP via Drop impl
}
```

### Module 2 — server-side WebSocket upgrade in `http.serve`

Extend `http.serve`'s response shape:

```lua
-- existing shapes (unchanged):
return { status = 200, body = ..., headers = ... }   -- normal HTTP
return { status = 200, json = ... }                   -- shorthand
return { status = 200, sse = function(send) ... end } -- SSE stream

-- NEW shape:
return { status = 101, ws = function(conn) ... end }  -- websocket upgrade
-- (status omitted: defaults to 101)
```

When the response table has a `ws` key, http.serve:

1. Validates the request has the WebSocket upgrade headers (`Upgrade: websocket`,
   `Connection: Upgrade`, `Sec-WebSocket-Key`, `Sec-WebSocket-Version: 13`)
2. Builds the `Sec-WebSocket-Accept` reply
3. Sends the 101 Switching Protocols response
4. Hands the upgraded stream to
   `tokio_tungstenite::WebSocketStream::from_raw_socket(stream, Server, None)`
5. Calls `function(conn)` with a `WsServerConn` UserData — async-aware via
   `lua.create_async_function`

#### `WsServerConn` UserData methods

```lua
local msg = conn:read()
-- Returns:
--   string      — next message (binary frames already decoded as raw bytes)
--   nil         — peer closed cleanly OR connection error (check conn:is_closed())
-- Coroutine-yields until next frame.

conn:write(data, opts?)
-- opts.binary = true  → binary frame (default: text frame)
-- Returns nil on success, errors on broken connection.
-- Coroutine-yields until frame is flushed.

conn:close(code?, reason?)
-- code default = 1000 (normal). reason default = "".
-- Sends a Close frame, waits for peer ack with 5 s timeout.

conn:is_closed() -> bool
conn.peer_addr -> string  -- e.g. "10.0.0.1:54321"
```

#### Implementation notes

- **`http.serve` runs on hyper directly** (`hyper::server::conn::http1`, see
  `crates/assay/src/lua/builtins/http.rs`), not axum. Do the upgrade by hand: validate the request
  headers, build the `Sec-WebSocket-Accept` reply, return a 101 response whose body is empty, then
  `hyper::upgrade::on(req)` to recover the underlying TCP stream and hand it to
  `tokio_tungstenite::WebSocketStream::from_raw_socket(stream, Role::Server, None)`.
- The 101 response is built synchronously inside `lua_response_to_http`, but the
  `ws =
  function(conn)` Lua callback runs after the upgrade completes — same
  `tokio::task::spawn_local` pattern the existing `sse =` path already uses.
- Backpressure: tungstenite's stream is naturally backpressured — `write()` blocks until the frame
  is sent. Keep that semantic.
- Concurrency: each connection runs in its own task, isolated from other handlers.
- Ping/pong: tungstenite handles automatically; expose via a small `conn:set_ping_interval(secs)`
  later if needed (not in v0.15.1).

#### Tests

-
  1. Server `/echo` handler: read→write back. Client connects, sends "hello", reads back "hello".
-
  2. Binary frames round-trip.
-
  3. Server closes via `conn:close(1000, "bye")`; client receives close frame.
-
  4. Concurrent: 5 clients, each gets independent reads.
-
  5. Backpressure: server pauses reading; client sends 100 frames; client write doesn't error
     (tungstenite buffers in TCP).

### Optional Module 3 — `assay.shell` Lua stdlib (umbrella)

A 50-line Lua wrapper that takes a websocket conn + a target command and bridges them, including
JSON-encoded resize control messages on a side channel. Lets a consumer write:

```lua
local shell = require("assay.shell")
shell.bridge(conn, {
  cmd  = "machinectl",
  args = { "shell", machine_name },
  cols = 80, rows = 24,
})
```

Resize protocol: client sends a special text frame `{"resize":{"cols":N,"rows":M}}`; the bridge
calls `pty:resize(N, M)`. Other text frames (and all binary frames) pass through to the PTY as raw
bytes.

This module is shippable without changing v0.15.1's surface — it's pure Lua on top of
`process.spawn_pty` + `ws server`.

## Implementation phases

Three independently-shippable commits, all on the v0.15.1 branch.

### Phase 0 — `process.spawn_pty` builtin

- Add `nix = "0.30"` to `crates/assay/Cargo.toml` under
  `[target.'cfg(any(target_os = "linux", target_os = "macos"))'.dependencies]` with the `term`,
  `process`, `signal`, and `fs` features.
- New mod inside `crates/assay/src/lua/builtins/process.rs` (or sibling file `process_pty.rs`):
  `PtyHandle` UserData, `spawn_pty_impl(opts) -> PtyHandle`.
- Tests as listed above. Use `tempfile::NamedTempFile` for fixture scripts.
- `cargo clippy --lib --all-targets -- -D warnings` green.

### Phase 1 — server-side WebSocket upgrade in `http.serve`

- Edit `crates/assay/src/lua/builtins/http.rs` to detect `ws =` in handler responses, perform a
  manual hyper upgrade (Sec-WebSocket-Accept handshake + `hyper::upgrade::on`), hand a
  `WsServerConn` UserData to the function.
- Define `WsServerConn` in `crates/assay/src/lua/builtins/ws.rs` (alongside the existing client
  `WsConn`). Reuse the SinkExt/StreamExt pattern; the underlying stream type is
  `hyper_util::rt::TokioIo<hyper::upgrade::Upgraded>` rather than `MaybeTlsStream<TcpStream>`.
- Need a small base64 + sha1 over the `Sec-WebSocket-Key` to compute `Sec-WebSocket-Accept` —
  `data-encoding` is already a dep; add `sha1 = "0.10"` (sha1 is required by the WS spec).
- Tests as listed above using `tokio_tungstenite::connect_async` to round-trip against a server
  bound to port 0.

### Phase 2 — `assay.shell` umbrella stdlib + docs

- `crates/assay/stdlib/shell.lua` (~50 LOC).
- `docs/modules/shell.md` reference page (or extend existing `process.md` / `http.md`).
- README + CHANGELOG v0.15.1 entries (append to the existing "Added" sections).
- One worked example in `examples/shell-server.lua` showing a complete browser-shell endpoint in ~40
  lines.

## Migration / compat

Purely additive. No breaking changes.

- `process.spawn`, `process.list`, etc. unchanged.
- `http.serve`'s existing `body` / `json` / `sse` response shapes unchanged. `ws` is a new option
  key.
- `ws` Lua client builtin unchanged. New `WsServerConn` UserData lives in the same module but is a
  distinct type.

## Public motivation (for CHANGELOG / blog)

> v0.15.1 brings native browser-shell capability to assay scripts. New `process.spawn_pty` allocates
> a fresh PTY, attaches a child process, exposes read/write/resize/close on a handle. New
> `{ws = function(conn) ... end}` response in `http.serve` handles the WebSocket server upgrade and
> gives Lua a duplex stream. The combination — plus the new `assay.shell` umbrella — lets a single
> static binary host an in-browser xterm.js terminal connected to any command (machinectl shell,
> kubectl exec, ssh, plain bash) without subprocess sidecars or external WebSocket servers.

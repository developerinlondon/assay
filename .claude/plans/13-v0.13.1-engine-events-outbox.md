# v0.13.1 — Engine Events Outbox (Durable Realtime, Backend-Agnostic)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps in each
> phase file use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the trigger-driven PG-only push stream + lossy in-memory SSE with a Rust-managed,
backend-agnostic CDC outbox (`engine_events`) that delivers durable realtime events to the
scheduler, task workers, and dashboard SSE clients, with cursor-based replay on reconnect and
cross-node coordination via a single PG `NOTIFY` channel per namespace.

**Architecture:** All state-mutating store methods emit typed events via a typed `WorkflowEventBus`
that wraps a generic `EngineEventBus` trait. Subscribers — scheduler, workers, SSE — read from a
shared per-node `tokio::broadcast`, fed by (a) direct local emits for same-node writes and (b) a
single `PgListener` that translates cross-node `NOTIFY` bumps into local fan-out. SQLite runs
single-process with in-process broadcast only; PG supports multi-node. Triggers are removed; the 15s
scheduler-poll fallback is removed; the in-memory `sse_tx`/`engine_tx` duplication is merged into
the EventBus.

**Tech Stack:** Rust 2024 edition (1.85+), sqlx 0.8 (PostgreSQL 18 + SQLite 3.35+), tokio 1, axum
0.8, async-trait 0.1, tokio-stream, serde 1 / serde_json 1, thiserror 2, anyhow 1,
futures-util/futures-core 0.3, async-stream 0.3. No new third-party deps beyond async-trait
(required for `Arc<dyn EngineEventBus>` — the trait is not RPITIT because we deliberately want dyn
dispatch at the engine-state boundary).

**Release target:** v0.13.1 umbrella, with per-crate bumps: `assay-v0.13.1`, `assay-engine-v0.1.1`,
`assay-workflow-v0.2.1`, `assay-domain-v0.1.1`. No changes to `assay-lua`, `assay-dashboard`,
`assay-auth` (no auth content in v0.13.1).

**Development mode:** This is active development on the engine (pre-1.0). Plan intentionally omits a
migration guide — consumers roll with it at each release. CHANGELOG entry suffices as release notes.

## Scope (v0.13.1)

- `engine_events` table on PG + SQLite, created via the standard startup SCHEMA path.
- `EngineEventBus` trait in `assay-domain` + `PgEngineEventBus` + `SqliteEngineEventBus`
  implementations.
- `WorkflowEventBus` typed wrapper in `assay-workflow` with a `WorkflowEvent` enum covering every
  state transition currently emitted via `ctx.broadcast(...)` plus the dispatch-wake-up +
  activity-inserted signals currently carried by `pg_notify`.
- Rewrite the scheduler wake-up path, task-dispatcher wake-up, and SSE surface to consume from
  `WorkflowEventBus`.
- Rewrite `GET /api/v1/events/stream` to honour `Last-Event-ID`, support
  `?ns=&subsystem=&workflow_id=&kind=` filters, return HTTP `410 Gone` on pre-retention gaps, and
  force-close on `broadcast::RecvError::Lagged`.
- Remove the `assay_runnable_<ns>` / `assay_task_<queue>` PL/pgSQL triggers entirely. Remove
  `TRIGGER_DDL` block.
- Remove the 15s periodic scan from the scheduler wake-up loop.
- Add a one-hour `tokio::time::interval` cleanup task that prunes `engine_events` older than the
  configured TTL.

## Out of scope (deferred)

- `AuthEventBus` — scaffold only; real events land in v0.14.0 with auth primitives.
- `SecretsEventBus`, `SystemScriptsEventBus` — not built, named as future subsystems only.
- Engine-scoped Lua system scripts (dashboard-scheduled Rust-fn-backed Lua) — v0.15+ idea, see
  "Future work" section. Current `assay-lua` (pure HTTP-client runtime) is unchanged.
- `postgres_changes`-style logical replication — explicitly rejected for slot-wedge ops risk.
- Migration doc — active development, consumers roll with it.

## Architecture at a glance

### Current (v0.13.0) — two incoherent paths, in-memory SSE loses data

```
            Rust store methods
                  │
                  ▼
        ┌──── workflows table ─────┐
        │                          │
   trigger                         │
 (PL/pgSQL)                        ▼
        │                   ctx.broadcast(...)
        │                          │
 pg_notify(                        │
   assay_runnable_<ns>,            │
   assay_task_<queue>)             │
        │                          ▼
        ▼                   tokio::broadcast(1024)
PgListener                         │
(per subscribe call)               ▼
        │                   BroadcastStream → SSE
        ▼                          │
Scheduler loop                     ▼
select! {                     Browser client
  notify | sleep(15s) scan   (NO replay — laptop sleeps
}                             30 s = silent loss)
```

### Proposed (v0.13.1) — one Rust-managed path, durable + cross-node

```
             Rust store methods
                       │
                       ▼
           ┌──────── BEGIN TX ────────┐
           │  INSERT workflows (...)  │
           │  bus.publish_committed(e)│
           │    └─ INSERT engine_events RETURNING id
           │    └─ [PG] pg_notify('assay_engine_events_<ns>', id)
           │  COMMIT                  │
           └────────────┬─────────────┘
                        │ (NOTIFY fires at commit — atomic)
                        ▼
            bus.local.send(event)       ← same-node fast path
                        │
        ┌───────────────┼─────────────────────┐
        ▼               ▼                     ▼
 ┌──────────┐   ┌──────────────┐   ┌───────────────────┐
 │scheduler │   │task workers  │   │SSE /events/stream │
 │.filter() │   │.filter()     │   │Last-Event-ID +    │
 └──────────┘   └──────────────┘   │server-side filter │
                                   └───────────────────┘

Remote node (multi-node PG):
 PgListener (TCP keepalive 30/10/3)
        │
 NOTIFY id → SELECT * FROM engine_events WHERE id = $1
        │
        ▼
 bus.local.send(event)  on that node → same fan-out
```

### Subsystem layering (Option 3: shared infra, typed wrappers)

```
┌─────────────────── assay-domain ───────────────────┐
│  trait EngineEventBus                              │
│  struct Event { id, ts, ns, subsystem, kind,       │
│                 payload: serde_json::Value }       │
│  impl PgEngineEventBus / SqliteEngineEventBus      │
│  ── ONE engine_events table                        │
│  ── ONE channel per ns                             │
│  ── ONE PgListener per node                        │
│  ── ONE tokio::broadcast per node                  │
└───────────┬──────────────────┬─────────────────────┘
            │                  │
            ▼                  ▼
┌──────────────────┐  ┌──────────────────────────────┐
│ assay-workflow   │  │ assay-auth (v0.14.0)         │
│ WorkflowEvent    │  │ AuthEvent  (scaffold only)   │
│ WorkflowEventBus │  │ AuthEventBus (scaffold only) │
└──────────────────┘  └──────────────────────────────┘
```

## Decisions locked (do not re-litigate)

| #  | Decision                                                                                                                  | Rationale                                                                               |
| -- | ------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| 1  | **Rust-managed CDC** — all emission via `EngineEventBus::publish_committed`; no PL/pgSQL triggers                         | Backend-agnostic; single source of truth at the call site; typed events; testable       |
| 2  | **Option 3 layering** — shared `engine_events` table + trait, typed per-subsystem wrappers                                | Matches "one process, fewer moving parts" while keeping crate boundaries                |
| 3  | **Typed event enum** — each subsystem owns its `FooEvent` enum; serialised as JSON into `engine_events.payload`           | Small payloads, testable predicates, no `to_jsonb(NEW)` bloat                           |
| 4  | **TCP keepalive on PgListener** — no app-level heartbeat, no polling                                                      | Kernel detects dead connections in ~60s; sqlx auto-reconnects; cursor replay covers gap |
| 5  | **Remove 15s periodic scheduler scan**                                                                                    | Durable outbox + cursor replay makes it redundant                                       |
| 6  | **Engine housekeeping = more `tokio::spawn`** — no new cron subsystem                                                     | Matches existing pattern (scheduler, timer_poller, health, dispatch_recovery)           |
| 7  | **Retention 3 days default**, configurable per subsystem via `EngineConfig`                                               | Matches Supabase `realtime.messages`; sensible upper bound for replay-on-reconnect      |
| 8  | **SSE subscription = combined query params** — `?ns=`, `?subsystem=` (repeatable), `?workflow_id=`, `?kind=` (repeatable) | Covers firehose + per-resource + per-subsystem as one endpoint                          |
| 9  | **Pre-retention gap → HTTP 410 Gone**                                                                                     | Forces snapshot reload when cursor is past TTL; standard SSE practice                   |
| 10 | **Lagged client → force-close SSE** with structured error; client reconnects with cursor replay                           | Avoids silent corruption; reuses the same replay path                                   |
| 11 | **Monotonic `BIGINT` id** from PG sequence / SQLite AUTOINCREMENT as the cursor                                           | Single-source-of-truth ordering; cheap; `Last-Event-ID` compatible                      |
| 12 | **No new deps beyond `async-trait`** — all other crates already in workspace                                              | Zero supply-chain surface added                                                         |
| 13 | **SQLite: single-process by design** — in-process broadcast only, no LISTEN equivalent                                    | Per Q1 answer; PG is the multi-node story                                               |
| 14 | **Active-dev mode** — no migration guide, CHANGELOG-only release notes                                                    | Pre-1.0 engine; consumers roll with each bump                                           |

## Rollout order (branch → main)

1. Branch cut: `feature/0.13.1-engine-events-outbox` from `main` ✓
2. **Phase 1 (13a)**: trait + types — additive, no behaviour change.
3. **Phase 2 (13b)** + **Phase 3 (13c)**: backend impls + DDL — adds `engine_events` table on fresh
   boots; existing tests untouched.
4. **Phase 4 (13d)**: `WorkflowEventBus` wrapper — additive.
5. **Phase 5 (13d)**: cutover emits — `ctx.broadcast` → `self.emit`. Legacy triggers + `subscribe_*`
   still present; new path exercised in parallel.
6. **Phase 6 (13e)**: flip scheduler to consume from bus; delete triggers + `subscribe_*`. This is
   the "no going back" commit — requires prior phases green.
7. **Phase 7 (13f)**: SSE rewrite — dashboard starts seeing the new payload; old shape is a subset.
8. **Phase 8 (13g)**: cleanup loop + retention config.
9. **Phase 9 (13g)**: TCP keepalive on `PgListener`.
10. **Phase 10 (13g)**: CHANGELOG + per-crate version bumps + draft PR.

Each phase is independently buildable and testable. If CI fails on any, revert that phase alone.

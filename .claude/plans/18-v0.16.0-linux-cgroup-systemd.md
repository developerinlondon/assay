# Plan 11: v0.16.0 — `linux` / `cgroup` / `systemd` builtins + `assay.cron` stdlib

## Problem

Operator-facing Lua scripts that monitor or control a Linux host today have two unappealing options:

1. **Shell out** to `systemctl`, `machinectl`, `journalctl`, `crontab`, `ip`, etc. via the existing
   `shell.exec` builtin. Each call is a fork+exec+text-parse round-trip; on a 60-unit host a single
   `systemctl list-units --output=json` takes 50–100 ms. Output formats also vary across systemd
   versions.
2. **Parse `/proc` / `/sys/fs/cgroup` from Lua**, which is fast (single-digit microseconds per file)
   but tedious — every consumer reinvents `/proc/stat` deltas, `/proc/meminfo` field extraction,
   `cgroup.cpu_stat` parsing, etc.

The result is that "watch CPU + memory of every nspawn machine on this host" — a one-liner in any Go
or Rust program with a procfs or zbus dependency — takes 200+ lines of fragile Lua per consumer to
reproduce, and runs an order of magnitude slower if it goes via shell.

Assay positions itself as the static-binary swiss-army knife for ops automation. The Linux
observability and control surface is a primary target use case, but today the runtime gives
consumers no native primitives for it.

## Goal

Ship four new modules in v0.16.0 so a Lua script can pull host metrics, walk container cgroups, talk
to systemd, and inspect cron schedules without forking subprocesses or hand-writing `/proc` parsers:

- **`linux`** (Rust builtin) — `/proc` and `/sys/fs/...` readers, returning structured tables for
  CPU, memory, load, uptime, network, disk I/O, and per-PID stats.
- **`cgroup`** (Rust builtin) — cgroup v2 unified-hierarchy readers for any cgroup path: `cpu.stat`,
  `memory.{current,max,peak,swap.*}`, `io.stat`, `pids.{current,max}`, `cgroup.procs`.
- **`systemd`** (Rust builtin, D-Bus client) — async `zbus` connection to `org.freedesktop.systemd1`
  and `org.freedesktop.machine1`. Lists units, lists timers, queries unit status, sends
  start/stop/restart/reload jobs, lists machines, sends machine lifecycle commands, streams the
  journal via `sd-journal`.
- **`assay.cron`** (Lua stdlib) — file-walk parsers for `/etc/crontab`, `/etc/cron.d/*`,
  `/etc/cron.{daily,hourly,weekly,monthly}/*`, `/var/spool/cron/crontabs/*`, plus a thin passthrough
  to `systemd.list_timers()` for a unified scheduled-job view.

Native everywhere it matters; no shell-outs in the hot path.

## Non-goals

- **Cgroup v1 support.** Cgroup v2 unified hierarchy is the floor — Ubuntu 22.04+, Fedora 31+,
  Debian bookworm+, RHEL 9+ all default to it. v1 / hybrid layouts return `cgroup.version() == "v1"`
  and callers can refuse. Adding v1 readers triples the surface area for declining real-world need.
- **Replacing `shell` and `process`.** `shell.exec` and `process.spawn` stay. Some operators will
  still want raw text from `iostat`, `nethogs`, custom scripts. The new modules just remove the need
  to shell out for the standard things.
- **Replacing `assay.k8s`.** Container introspection here is _systemd-machined_ containers, not
  Kubernetes pods. The two stay separate; a consumer that runs k3s inside an nspawn container uses
  both modules side-by-side.
- **Cross-platform parity.** This release is Linux-only for the new modules. macOS / Windows builds
  compile the modules with stubs that error at runtime (`linux.cpu_stat: Linux only`). Same posture
  the existing `process.list` builtin already takes.
- **HTTP API surface.** None of these modules ship an `assay-engine` HTTP route. They are pure Lua
  primitives.

## Design

### Module 1 — `linux` (Rust builtin)

`crates/assay/src/lua/builtins/linux.rs`. Backed by the [`procfs`](https://crates.io/crates/procfs)
crate (MIT, ~5 k stars, in active use by `bottom`, `nu_plugin_inc`, others). Linux-only — module
table is empty on other platforms, every function returns
`mlua::Error::runtime("linux: Linux only")`.

Lua surface:

```lua
linux.kernel()              -- {version, hostname, os_release, btime}
linux.uptime()              -- {uptime_secs, idle_secs}
linux.loadavg()             -- {one, five, fifteen, running, total, last_pid}

linux.cpu_stat()            -- /proc/stat aggregate row, in jiffies:
                            -- {user, nice, system, idle, iowait, irq, softirq,
                            --  steal, guest, guest_nice}
linux.cpu_stat_per_core()   -- [{cpu=0, user=…, …}, {cpu=1, …}, …]
linux.cpu_percent(prev, curr)
                            -- helper: takes two snapshots from cpu_stat(),
                            -- returns {total_pct, per_core={…}}

linux.meminfo()             -- /proc/meminfo as {total, free, available,
                            --  buffers, cached, swap_total, swap_free,
                            --  shmem, slab, …} — bytes (not kB)

linux.netdev()              -- /proc/net/dev:
                            -- [{name, rx_bytes, rx_packets, rx_errs, rx_drop,
                            --   tx_bytes, tx_packets, tx_errs, tx_drop}, …]

linux.diskstats()           -- /proc/diskstats:
                            -- [{device, reads, reads_merged, sectors_read,
                            --   ms_read, writes, …, in_progress, ms_io}, …]

linux.proc_stat(pid)        -- /proc/<pid>/stat parsed:
                            -- {pid, comm, state, ppid, pgrp, session,
                            --  utime, stime, vsize, rss_pages, num_threads,
                            --  starttime, …}
linux.proc_status(pid)      -- /proc/<pid>/status fields:
                            -- {name, uid, gid, pid, ppid, vm_rss, vm_size,
                            --  threads, …}
```

`cpu_percent(prev, curr)` is a Lua-level convenience computed from two `cpu_stat()` snapshots —
delta jiffies divided by total delta gives a percent, no kernel call. Keeps the API closer to "just
call `cpu_percent` on a 5 s loop" without baking sleep into the builtin.

Numbers come back as Lua integers where they fit and floats otherwise (`mlua` handles this). All
`bytes` fields are bytes — we do the kB→byte conversion in Rust so callers don't have to.

### Module 2 — `cgroup` (Rust builtin)

`crates/assay/src/lua/builtins/cgroup.rs`. Pure file-read implementation; no crate dependency.
Detects `cgroup2fs` at `/sys/fs/cgroup` via `statfs` and refuses to operate on v1.

```lua
cgroup.version()                   -- "v2" | "v1" | "hybrid"
cgroup.list(slice_path)            -- ["systemd-nspawn@app-1.service",
                                   --  "user.slice", …]  — names of child cgroup dirs
cgroup.cpu_stat(path)              -- {usage_usec, user_usec, system_usec,
                                   --  nr_periods, nr_throttled, throttled_usec,
                                   --  nr_bursts, burst_usec}
cgroup.memory(path)                -- {current, max, swap_current, swap_max,
                                   --  peak, low, high, oom_kill, oom}
cgroup.io(path)                    -- io.stat parsed: [{device, rbytes, wbytes,
                                   --                    rios, wios, dbytes, dios}, …]
cgroup.pids(path)                  -- {current, max}
cgroup.procs(path)                 -- [pid, pid, …]   — cgroup.procs file
```

`path` is an absolute filesystem path under `/sys/fs/cgroup/` so callers can read any cgroup, not
just `machine.slice`. `cgroup.list("/sys/fs/cgroup/machine.slice")` returns every container's
service unit; `cgroup.list("/sys/fs/cgroup/user.slice")` returns per-user slices; etc.
Path-traversal protection: every input path is canonicalised and required to start with
`/sys/fs/cgroup/`.

`memory.max` returns `nil` when the file contains the literal string `max` (cgroup v2's "unlimited"
sentinel) — the integer-vs-string ambiguity is hidden from Lua callers, who can check
`if not mem.max then ...`.

### Module 3 — `systemd` (Rust builtin, D-Bus)

`crates/assay/src/lua/builtins/systemd.rs`. Backed by [`zbus`](https://crates.io/crates/zbus) (4.x;
async, uses `tokio` runtime which assay already runs). One persistent system-bus connection per Lua
VM, lazily opened on first call. Methods that send unit lifecycle jobs use
`org.freedesktop.systemd1.Manager.StartUnit("...service", "replace")` etc., returning the job object
path. Machine methods talk to `org.freedesktop.machine1.Manager`. Journal access uses `sd-journal`
(`libsystemd-sys`) — D-Bus does not expose journal entries.

```lua
-- Units
systemd.list_units(filter?)        -- [{name, load, active, sub, description,
                                   --   following, unit_object_path}, …]
                                   -- filter: "*.service", "*.timer", or nil
systemd.unit_status(name)          -- full unit dict (same shape as `systemctl
                                   -- show --output=json` but typed):
                                   -- {load_state, active_state, sub_state,
                                   --  description, since, main_pid,
                                   --  exec_main_status, fragment_path,
                                   --  drop_in_paths={...}, …}
systemd.is_active(name)            -- boolean
systemd.list_timers()              -- [{unit, next_elapse_realtime,
                                   --   last_trigger_realtime, passed,
                                   --   activates}, …]

systemd.start(name) / stop(name) / restart(name) / reload(name)
                                   -- returns the job object path
                                   -- (string, e.g. "/org/freedesktop/systemd1/job/42")

-- Machines (org.freedesktop.machine1)
systemd.list_machines()            -- [{name, class, service, leader_pid,
                                   --   addresses=[…], root_directory}, …]
                                   -- class is "container" / "vm" / "host"
systemd.machine_status(name)       -- full machine dict (timestamp, leader,
                                   --                    network namespace, …)
systemd.machine_start(name)
systemd.machine_poweroff(name)
systemd.machine_reboot(name)
systemd.machine_terminate(name)

-- Journal
systemd.journal(opts)              -- one-shot read of last N entries:
                                   -- opts = {unit?, machine?, since?, until?,
                                   --         lines?=200, priority?=7, follow?=false}
                                   -- returns [{ts, hostname, unit, message,
                                   --           priority, transport}, …]
systemd.journal_follow(opts, fn)   -- streaming variant. fn is called once per
                                   -- entry with the same table shape.
                                   -- Returns a cancel handle; call handle:close()
                                   -- to stop. Uses sd_journal_wait under the hood
                                   -- so it's not a busy loop.
```

The journal streamer lands on `process.spawn`'s pattern: it returns a handle whose `:close()`
deregisters from the runtime. Existing SSE consumers in operator dashboards become a single
`for entry in iter do send(entry) end` loop without subprocess management.

D-Bus connection lifetime: opened on first call, kept until VM teardown. Lua VMs are short-lived
(per-script in CLI mode; per-request in `http.serve` is _not_ a thing — `http.serve` shares one VM).
A single bus connection per VM is the right grain — connection cost (~5 ms) amortises.

Permissions: methods that send jobs (`start`, `stop`, machine lifecycle) require the calling process
to have polkit authorisation for the relevant D-Bus interface. Documentation in
`docs/modules/systemd.md` lists the polkit actions consumers need to allow.

### Module 4 — `assay.cron` (Lua stdlib)

`crates/assay/stdlib/cron.lua`. Pure-Lua because the cost is dominated by file reads, not parsing,
and a Rust implementation would lock the parsing rules in. Auto-loaded via the existing
`include_dir!` discovery in `crates/assay/src/lua/mod.rs`.

```lua
local cron = require("assay.cron")

cron.system_crontab()        -- parses /etc/crontab + /etc/cron.d/*
                             -- → [{source, line, schedule, user, command}, …]
                             -- where schedule is the raw 5/6-field cron string
cron.user_crontabs()         -- {user = parsed_crontab, …} from
                             -- /var/spool/cron/crontabs/* (root-readable)
cron.daily_dropins()         -- {hourly={…}, daily={…}, weekly={…}, monthly={…}}
                             -- listings of /etc/cron.{hourly,daily,weekly,monthly}/
cron.timers()                -- thin wrapper: returns systemd.list_timers()
                             -- (errors with a hint if the systemd builtin is
                             --  unavailable — e.g. on macOS)
cron.all()                   -- unified [{source, schedule, command, user,
                             --           next_fire?, last_fire?}, …]
                             -- merging all four sources above.
                             -- next_fire/last_fire are present only for systemd
                             -- timers (they're computed by systemd, not knowable
                             -- for crontab entries without re-implementing cron's
                             -- scheduler).
```

Doc convention follows `healthcheck.lua` — `--- @module`, `--- @description`, `--- @keywords`,
`--- @quickref` lines so the discovery layer indexes the module correctly.

## Implementation phases

Five independently-shippable, independently-reviewable PRs. Each ends green on
`cargo clippy
--all-targets -- -D warnings` and `cargo test`.

### Phase 0 — Skeleton + smoke

Branch already exists: `feature/v0.16.0-linux-sysd-builtins`.

- Stub registration only: `linux`, `cgroup`, `systemd` modules registered, every function returns
  `mlua::Error::runtime("not yet implemented")`.
- One integration test per module that loads the binary and asserts the table exists with the right
  method names.
- CHANGELOG entry under `changelog/v0.16.0.md` (per the plan-09 layout) listing the four pending
  modules as `### Added` placeholders.

### Phase 1 — `linux` builtin

- Add `procfs = "0.16"` to `crates/assay/Cargo.toml`.
- Implement every function listed in §Design. Reference: existing `disk.rs` and `process.rs` for the
  table-build pattern.
- Tests: round-trip a known synthetic `/proc/stat` string through the parser; live-system test on
  Linux CI runners that asserts `linux.meminfo().total > 0`.
- Doc: `docs/modules/linux.md` with one example per function.

### Phase 2 — `cgroup` builtin

- No new crate dependency — pure stdlib `std::fs` reads.
- v2 detection via `statfs` (existing `disk.rs` already shows the `libc::statvfs` call shape).
- Path canonicalisation + `/sys/fs/cgroup/` prefix check before every read.
- Tests: synthetic fixture cgroup tree under `target/test-fixtures/cgroup/...` for the parsers;
  on-host smoke tests gated by a `cgroup_v2` env feature flag.

### Phase 3 — `systemd` builtin

- Add `zbus = "4"` and `libsystemd-sys = "0.5"` (or `systemd-journal-logger`'s journal API) to
  `crates/assay/Cargo.toml`, behind a `linux-only` feature on Linux targets.
- Implement units, machines, then journal. Each as its own commit.
- Tests:
  - `list_units()` on the CI runner returns at least `init.scope`.
  - `is_active("ssh.service")` returns a boolean (don't assert true; CI runners may not run sshd).
  - Journal one-shot returns last `N` entries.
  - Journal follow: spawn a background `logger -t test "$RANDOM"` and assert the streamer surfaces
    it.
- Doc: `docs/modules/systemd.md` including the polkit action names, plus a worked example for "tail
  the journal of an nspawn container into an SSE handler".

### Phase 4 — `assay.cron` stdlib

- Pure Lua, ~150 LOC. Crontab parser handles 5-field, 6-field (with seconds), and `@reboot` /
  `@daily` / `@hourly` shorthand.
- Tests: fixture `/etc/crontab` and `/etc/cron.d/sysstat` examples in
  `crates/assay/stdlib/cron_spec.lua` (busted-style, run by `cargo test`'s Lua harness).
- Doc strings (`--- @module assay.cron` etc.) so the discovery layer picks it up.

### Phase 5 — CHANGELOG, README, site

- Promote the placeholders in `changelog/v0.16.0.md` to real entries with examples.
- New rows in `README.md`'s stdlib + builtins tables.
- New section in `site/modules.html` for each of `linux`, `cgroup`, `systemd`, `assay.cron`.
- `docs/migration-to-0.16.0.md` with the high-level summary.

## Migration / compat

No breaking changes — every API in this plan is **additive**.

- `process.list` and `shell.exec` stay; consumers that already use them keep working.
- v1 cgroup hosts get `cgroup.version() == "v1"` and explicit errors from the cgroup readers — the
  alternative (silent failure) is worse.
- macOS / Windows builds compile the new modules but every function returns runtime errors. CI
  gating will catch any consumer that expected cross-platform behaviour.
- `docs/migration-to-0.16.0.md` explains for the rare reader that wants to swap their existing
  `shell.exec("systemctl ...")` calls for the native API.

## Public motivation (for CHANGELOG / blog)

> v0.16.0 brings a native Linux observability and systemd control surface to assay scripts. `linux`
> reads `/proc` directly, `cgroup` walks the unified-hierarchy v2 tree without shelling out,
> `systemd` talks D-Bus to query units and machines and stream the journal, and `assay.cron` rolls
> up scheduled jobs across systemd timers and traditional crontabs. Operator dashboards and
> health-check scripts that previously fork-and-parsed `systemctl`, `machinectl`, and `journalctl`
> ten times a refresh cycle now do the same work in single-digit milliseconds via persistent D-Bus
> connections and tmpfs reads.

# Plan 17 — assay-engine v0.3.0: safe module

**Status:** drafted, awaiting Phase 0 implementation **Successor of:** plan 12 (v0.2.0 auth + IdP)
**Target tag:** `assay-engine-v0.3.0` **Sibling crate:** `assay-safe` (new) at `crates/assay-safe/`

## Naming

The module is called **safe** — not "secrets" or "vault".

- "Secrets" leans corporate-ops; doesn't fit the personal-use side (passwords, SSH keys, notes,
  identity docs, software licenses, TOTP seeds, recovery codes, crypto wallet seeds).
- "Vault" collides with HashiCorp Vault and conflicts with the inner noun: a _vault_ is a 1pw-style
  container that lives _inside_ the safe.
- "Safe" reads naturally for both ops ("API key in the safe") and humans ("my passport is in the
  safe"). It frames the module as a place to keep valuables, which matches what people actually
  store in it.

The internal vocabulary keeps the 1pw term: a _vault_ (e.g. "Engineering team vault",
"alice-personal") is a 1pw-style container within the safe.

## Why this exists

assay-engine v0.2.x already covers identity, authz, workflows. The fourth thing every self-hosted
ops stack needs is a **safe** — the gap is currently filled by Vault (service secrets) plus
1Password / Bitwarden (human secrets). They overlap in capability, don't share identity, don't share
audit, run as separate stacks.

The unified-platform thesis: one binary, one auth model, one audit log, one dashboard. Add the safe
as the fourth module and the ~300 MB compressed footprint of (Vault + 1pw self-host) collapses into
the existing engine binary at +3 MB. Plus the entire Bitwarden client ecosystem (mobile / browser /
CLI / forks) becomes a day-1 human UX for free via the BW-protocol compat shim.

**First concrete consumer**: an existing in-house project that currently connects to AWS for service
credentials. Drop-in replacement target — they switch from rotating long-lived AWS access keys to
short-lived AssumeRole leases via assay-safe.

**Embedding angle**: jeebon (and any future assay-engine consumer) imports the engine crate with
`safe` enabled and gets ops + human secrets storage as a built-in product surface, no integration
work.

## Scope (what's IN)

### S1. KV v2 — versioned key/value store

HashiCorp Vault's KV v2 semantics: namespaced paths, soft-delete, version history, opt-in metadata.
Server-side encryption (per-record DEK wrapped by master KEK; engine can decrypt to serve).

- `PUT  /api/v1/safe/kv/{path}` — store new version
- `GET  /api/v1/safe/kv/{path}?version=N` — read (default latest)
- `LIST /api/v1/safe/kv/{prefix}` — list paths
- `DELETE /api/v1/safe/kv/{path}?version=N` — soft delete
- `POST /api/v1/safe/kv/{path}/destroy?version=N` — hard delete
- `POST /api/v1/safe/kv/{path}/undelete?version=N`

Storage: `safe.safe_kv` (versioned blobs) + `safe.safe_kv_meta` (latest version, custom md, deletion
state). KEK rotates via existing `auth.jwks_keys`-shaped key registry.

### S2. transit — encrypt/decrypt without exposing the key

Vault's transit semantics: operators register a key, clients call encrypt/decrypt without ever
seeing the raw key. Useful for application-layer encryption (PII fields, etc.).

- `POST /api/v1/safe/transit/keys/{name}` — create key
- `POST /api/v1/safe/transit/encrypt/{name}` — `{plaintext}` → `{ciphertext}` (versioned)
- `POST /api/v1/safe/transit/decrypt/{name}` — `{ciphertext}` → `{plaintext}` (any version)
- `POST /api/v1/safe/transit/keys/{name}/rotate` — bump version; old versions stay decryptable

Crypto: AES-256-GCM-SIV (deterministic, misuse-resistant). One key per name, multiple versions.

### S3. dynamic creds — short-lived service credentials

Four providers in v0.3.0, all native:

- **S3a Postgres**: pre-configured master role; `POST /api/v1/safe/dynamic/postgres/{role}/lease`
  issues a short-lived role with the configured grants, returns
  `{username, password, lease_id, expires_at}`. Lease renew + revoke.
- **S3b AWS IAM**: `sts:AssumeRole` with a configured ARN, returns
  `{access_key_id, secret_access_key, session_token, expires_at}`. Also supports
  `iam:CreateAccessKey` for the user-pool variant with auto-cleanup on lease expiry.
- **S3c GCP IAM**: service account impersonation via `iam.serviceAccounts.signJwt`
  - token exchange. Returns short-lived OAuth2 access tokens.
- **S3d Kubernetes**: ServiceAccount-token projection for the engine's own cluster. Returns a fresh
  projected token for a configured SA + audience.

Lease tracking in `safe.safe_leases`. Background task revokes on expiry. **Plugin trait**
`DynamicCredsProvider` is exposed so operators can register their own native providers (or
eventually hook in via Lua, but Lua-backed providers are deferred to v0.4.x since the existing
project needs the AWS path now).

### S4. vaults + sharing — 1pw-style team/personal storage

A _vault_ is a permission scope owned by a user or a Zanzibar object. Items inside a vault are E2E
encrypted with a per-vault wrapping key, which is encrypted to each member's public key (X25519).
Items are encrypted with the vault key (AES-256-GCM-SIV).

- `POST /api/v1/safe/vaults` — create vault
- `POST /api/v1/safe/vaults/{id}/items` — add item
- `GET  /api/v1/safe/vaults/{id}/items/{item}` — fetch ciphertext; client decrypts using their
  wrapped vault key
- `POST /api/v1/safe/vaults/{id}/members` — add member (writes a new envelope wrap)

Server is **zero-trust** for vault contents: cannot decrypt items, cannot decrypt vault keys. Only
sees ciphertext blobs and audit metadata.

Membership is Zanzibar tuples. Default schema ships:

```
namespace vault {
  relation owner:  user
  relation editor: user | team#member
  relation viewer: user | team#member | family#member
  relation member: editor | viewer | owner   # rolled-up
}
namespace team {
  relation member: user
  relation admin:  user
}
namespace family {
  relation member: user
}
```

This gives personal vaults, team vaults with auto-share via team membership, and family/household
vaults with the same shape. All expressible as Zanzibar tuples; the recursive-CTE walk that already
powers workflow access reuses without modification.

**Caveat**: Zanzibar can authorize _access_, but the per-member envelope-wrapped vault key must be
created for each new member. Server can't wrap (it doesn't have the vault key in plaintext). The
flow is: an admin's online client sees a new member → fetches their pubkey → wraps the vault key →
POSTs the envelope. New members see "pending access" if no admin is online; this is the 1pw /
Bitwarden model and there's no E2E-preserving way around it.

A "recovery delegate" / Shamir-secret-sharing pattern for offline admin-less wrap is **deferred to
v0.4.x**.

### S5. biscuit-share — assay-only differentiator

biscuit lets us mint a capability token granting narrow access to a specific item or path, with
on-token attenuation (time-bound, IP-bound, single-use). Verify offline. Vault and 1pw both require
server round-trips to revoke share links; biscuit can do expiry/IP scope without ever touching the
server.

- `POST /api/v1/safe/share` — `{item_id, ttl_secs, max_ip, ...}` → biscuit token
- `GET /safe/share/{token}` — public endpoint, validates the biscuit, returns the item if all
  caveats pass
- Server-side revocation list (by biscuit `key_id`) for the cases where attenuation isn't enough

This is the assay-distinct capability — Vault doesn't have it, 1pw doesn't have it. It's a real
reason to choose us.

### S6. Bitwarden-protocol compatibility shim

Existing Bitwarden mobile / browser / CLI clients are mature and multi-platform. Implement the
subset of their HTTP API (`/api/sync`, `/api/ciphers`, `/api/folders`,
`/api/identity/connect/token`) so users can point a stock Bitwarden client at `assay-engine` and get
the human-friendly UX for free.

Reference: `vaultwarden` does this exact thing in Rust; we can study their handlers as a template
(MIT-licensed, source readable at `/opt/forks/vaultwarden`).

Bitwarden also supports passkeys natively (both as stored items AND as vault-unlock); we get that
for free since `assay-auth/webauthn-rs` is already in tree. Same passkey that unlocks the dashboard
session unlocks safe access via session-bound key derivation (HKDF over session id + per-user salt).

In compat scope for v0.3.0:

- Login + session via Bitwarden's auth flow
- Cipher CRUD + sync
- Folder CRUD
- Passkeys as cipher-type items
- Bitwarden's two-step auth flow

Out of compat scope for v0.3.0:

- Send (biscuit-share is the assay-native equivalent and richer)
- Emergency Access (deferred)
- Device approval flows (deferred)
- Organizations (Bitwarden's term for teams; our Zanzibar team-vault model is the equivalent but not
  protocol-identical; deferred to v0.3.x)
- File attachments (deferred to v0.3.x)

### S7. Dashboard panes

New section in `assay-dashboard` mounted at `/safe/console`:

- **KV browser** — Vault-style tree of `safe/kv/...` (ops surface)
- **Transit keys** — list, rotate, view key versions (ops surface)
- **Dynamic leases** — current outstanding leases per provider, revoke
- **My vaults** — list of personal + shared vaults; click → items (humans surface)
- **Items** — login, password, SSH key, secure note, software license, identity doc;
  copy-to-clipboard, autofill hints
- **Share manager** — biscuit links you've minted, revoke, view caveats
- **Audit** — safe-scoped audit log (who, what, when — never _what content_)

Same pattern as auth-console: SPA, reads `/api/v1/modules` to render panes only when the `safe`
module is enabled.

The **primary human UX is Bitwarden mobile/browser/CLI** via the S6 compat shim — the dashboard
panes above are admin overlays for power users and operators. We don't try to compete with Bitwarden
on autofill or mobile UX.

### S8. Lua stdlib — `assay.safe.*`

Mirror of `assay.auth.*`. Programmatic API for scripts, agents, and workflow activities:

```lua
local safe = require("assay.safe")
local c = safe.client({ engine_url = "..." })

-- KV
c.kv:get("api/stripe")            --> { value, version, created_at }
c.kv:put("api/stripe", { ... })

-- Transit
c.transit:encrypt("logs", "...")  --> ciphertext
c.transit:decrypt("logs", ct)     --> plaintext

-- Dynamic creds
local lease = c.dynamic.aws:lease("ci-deploy", { ttl_secs = 1800 })
local lease = c.dynamic.postgres:lease("readonly")

-- Vaults
local vault = c.vaults:create({ name = "deploy", members = {"alice"} })
c.vaults:add_item(vault.id, { type = "password", ... })

-- Share
c.share:mint({ item_id = "...", ttl_secs = 3600, max_ip = "10.0.0.0/8" })
```

### Engine wiring

- New module entry in `engine.modules` (matching `auth`, `workflow`). Module name: `safe`.
- Cargo features: `safe-kv`, `safe-transit`, `safe-dynamic`, `safe-dynamic-postgres`,
  `safe-dynamic-aws`, `safe-dynamic-gcp`, `safe-dynamic-kubernetes`, `safe-vault`, `safe-share`,
  `safe-bitwarden-compat`. Default-on group `safe` enables KV + transit + vault + share + BW-compat.
  Dynamic-creds feature flags individually selectable so slim builds skip cloud SDKs they don't
  need.
- Schema namespace: `safe` (PG schema; SQLite attached file). Tables: `safe_kv`, `safe_kv_meta`,
  `safe_transit_keys`, `safe_transit_versions`, `safe_leases`, `safe_vaults`, `safe_vault_members`,
  `safe_items`, `safe_share_revoked`.
- `SafeCtx` composed into the central state struct, reuses the same backend pool (PG or SQLite) the
  rest of the engine uses.

### Crypto choices (locked)

| Concern                            | Primitive                                                         | Why                                                |
| ---------------------------------- | ----------------------------------------------------------------- | -------------------------------------------------- |
| KV at-rest encryption              | AES-256-GCM-SIV (per-record DEK)                                  | Misuse-resistant AEAD. RustCrypto.                 |
| Transit                            | AES-256-GCM-SIV with 96-bit nonce, version in AAD                 | Same primitive; nonce derived from version+counter |
| Vault wrap key (E2E)               | X25519 ECDH for sharing; AES-256-GCM-SIV item encryption          | Standard 1pw/Bitwarden shape                       |
| User keypair                       | X25519 (encryption) + Ed25519 (signing)                           | Already in tree via webauthn-rs / assay-auth       |
| Passkey-derived vault unlock       | HKDF over session-id + per-user salt + WebAuthn-attested material | No new flow — reuses session                       |
| Capability tokens (biscuit-share)  | biscuit (Datalog caveats)                                         | Already in tree, foundational in auth              |
| Master key (KEK)                   | KEK in `auth.jwks_keys`-shaped table, rotatable                   | Reuses key-registry mechanism                      |
| Bitwarden-compat client KDF        | PBKDF2-SHA256 / Argon2id (BW protocol)                            | Both already in tree                               |
| Bitwarden-compat cipher encryption | ChaCha20-Poly1305 (BW protocol)                                   | RustCrypto, ~50KB                                  |

**No custom protocols. No homebrew crypto.** Every primitive has a peer-reviewed implementation in
the Rust ecosystem.

## Phases (all in one feature branch, single PR per maintainer note)

### Phase 2 — vaults + sharing math (3-4 sessions)

- Vault create / member add / item add / item get.

- Default vault/team/family namespace schemas shipped with the module.
- Tests: create vault → add 3 members → each member can decrypt → 4th cannot → revoke member →
  previously-encrypted items stay readable but no new shares.
- Lua stdlib: `assay.safe.vaults`.
- Dashboard pane: My Vaults + Items.

### Phase 3 — biscuit-share (1-2 sessions)

- Mint short-lived capability tokens for an item or a path.
- Public-facing share endpoint validates biscuit caveats.
- Server-side revoke list (by biscuit key id).
- Tests: mint → fetch (allowed) → mint with TTL=1s, sleep 2s, fetch (denied); revoke key → fetch
  (denied); attenuate to specific IP and verify offline.
- Lua stdlib: `assay.safe.share`.
- Dashboard pane: Share manager.

### Phase 4 — dynamic creds (4-5 sessions)

- PG: master-role config, lease issuance, revoke, expiry sweeper.

- Tests: lease lifecycle for each provider against localstack / kind / fake_gcs_server; revoke fires

- Lua stdlib: `assay.safe.dynamic.{postgres,aws,gcp,kubernetes}`.

## What's explicitly OUT of scope

- **PKI / cert authority.** Vault has it. We don't. Use cert-manager.
- **ACME** as a backend.
- **Azure dynamic creds.** Three clouds (PG + AWS + GCP + K8s) cover the actual real-world use;
  Azure can land in v0.3.x or v0.4 based on demand.
- **SSH cert signing.**
- **Lua-backed dynamic-creds providers.** The plugin trait is Rust-only in v0.3.0; Lua-backed
  providers are v0.4+ once the trait surface is stable.
- **Cloud HSM / KMS integration** for the master KEK. Plaintext KEK-at-rest in DB for v0.3.0; HSM
  lands in v0.4.0.
- **Replication / multi-region.** Same posture as the rest of the engine — single-leader, follower
  replicas come later.
- **Bitwarden Send / Emergency Access / Organizations.** Send is largely covered by biscuit-share;
  Org is a different IAM model than ours; Emergency Access is a separate phase.
- **File attachments** (Bitwarden-compat). v0.3.x.
- **Mobile / browser frontend** (assay-native). BW-compat shim lets the existing clients work — we
  don't ship our own. CLI + browser extension may land in v0.4.x.
- **Recovery delegate / Shamir-secret-sharing** for offline admin envelope wrap. v0.4.x.
- **Rotation alerts / scheduled rotation.** Manual rotation only in v0.3.0; cron-based rotation
  rides on top of `assay-workflow` later.
- **Audit log forwarding** (S3, syslog). Surfaces in dashboard + Postgres only.
- **Compliance certifications** (SOC2 / FIPS / FedRAMP). Build the primitives, defer the audit.

## Open questions to resolve before Phase 1 commits

1. **Bitwarden client KDF**. BW supports both PBKDF2 and Argon2id; pick a default for new accounts
   (Argon2id matches what assay-auth already does for password storage; defaults align).
2. **Master KEK rotation cadence.** Reuse `auth.jwks_keys` rotation triggers, or independent?
   Probably independent — JWKS rotation is fast (issue new tokens), KEK rotation is slow (re-encrypt
   every DEK).
3. **biscuit issuer key handling.** Reuse the auth biscuit root key, or have a separate
   safe-biscuit-root? Separate keeps blast radius small if a key is compromised.
4. **AWS dynamic-creds first-customer integration.** The existing in-house project that connects to
   AWS — confirm exact endpoint shape before Phase 4 starts so the API matches their migration path.

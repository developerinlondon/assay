# Plan 17 — assay-engine v0.3.0: vault module

**Status:** locked, awaiting Phase 0 implementation **Successor of:** plan 12 (v0.2.0 auth + IdP)
**Target tag:** `assay-engine-v0.3.0` **New crate:** `assay-vault` at `crates/assay-vault/`
**Feature branch:** `feature/0.3.0-secrets` (branch name is historical; crate / module / API names
use `vault` throughout)

## Why this exists

assay-engine v0.2.x already covers identity, authz, workflows. The fourth thing every self-hosted
ops stack needs is a **vault** — the gap is currently filled by HashiCorp Vault (service secrets)
plus 1Password / Bitwarden (human secrets). They overlap in capability, don't share identity, don't
share audit, run as separate stacks.

The unified-platform thesis: one binary, one auth model, one audit log, one dashboard. Add the vault
as the fourth module and the ~300 MB compressed footprint of (Vault + 1pw self-host) collapses into
the existing engine binary at +1.7 MB. Plus the entire Bitwarden client ecosystem (mobile / browser
/ CLI / forks) becomes a day-1 human UX for free via the BW-protocol compat shim.

**First concrete consumer**: an existing in-house project that currently connects to AWS for service
credentials. Drop-in replacement target — they switch from rotating long-lived AWS access keys to
short-lived AssumeRole leases via assay-vault.

**Embedding angle**: jeebon (and any future assay-engine consumer) imports the engine crate with
`vault` enabled and gets ops + human secrets storage as a built-in product surface, no integration
work.

## Naming

The module is **vault** — Bitwarden-aligned, distinctive but recognisable, no inner-noun collision.

Internal model uses Bitwarden's vocabulary directly so the BW-compat shim is a field-for-field map,
not a translation layer:

| Concept        | Meaning                                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **vault**      | Per-user personal storage. Auto-created on signup. One per user. Holds personal items.                                         |
| **collection** | Shared container. Teams / families / orgs create collections, add members, add items. The 1pw "team vault" concept.            |
| **item**       | The leaf — login, password, SSH key, secure note, passkey, identity doc, software license. Lives in a vault OR a collection.   |
| **folder**     | Bitwarden-compat visual organization inside a collection (or a vault). Not an access boundary.                                 |
| **org**        | Top-level Zanzibar object scoping collections to a tenant.                                                                     |
| **kv_path**    | Server-side ops-secrets tree (Vault-equivalent — not E2E, server-decryptable). Arbitrary path hierarchy with prefix-based ACL. |

So: a user has their personal vault. Teams share via collections. Operators store service secrets in
the kv-path tree. Everything is items at the leaf, just with different encryption posture and
different access surface.

## Scope (locked) — what's IN v0.3.0

### S1. KV v2 (server-decryptable, ops surface)

Vault-equivalent. Path-tree storage, version history, soft-delete, custom metadata. Encrypted at
rest with a per-record DEK wrapped by the master KEK.

```
PUT  /api/v1/vault/kv/{path}                    # store new version
GET  /api/v1/vault/kv/{path}?version=N          # read (default latest)
LIST /api/v1/vault/kv/{prefix}                  # list paths
DELETE /api/v1/vault/kv/{path}?version=N        # soft delete
POST /api/v1/vault/kv/{path}/destroy?version=N  # hard delete
POST /api/v1/vault/kv/{path}/undelete?version=N
```

### S2. Transit (encrypt/decrypt without exposing the key)

Vault-equivalent. Operator registers a key; clients call encrypt / decrypt without seeing raw key
material. AES-256-GCM-SIV; rotate by bumping version (old versions stay decryptable).

```
POST   /api/v1/vault/transit/keys/{name}
POST   /api/v1/vault/transit/encrypt/{name}
POST   /api/v1/vault/transit/decrypt/{name}
POST   /api/v1/vault/transit/keys/{name}/rotate
```

### S3. Dynamic credentials (4 providers, minimal-SDK style)

Short-lived service credentials. Lease tracking in `vault.leases`; background sweeper revokes on
expiry. `DynamicCredsProvider` Rust trait is the extension point — additional providers (MySQL,
Mongo, Azure, etc.) implement the trait out-of-tree.

- **S3a Postgres** — pre-configured master role; `POST /vault/dynamic/postgres/{role}/lease` issues
  a short-lived role with the configured grants.
- **S3b AWS IAM** — `sts:AssumeRole` and `iam:CreateAccessKey` paths. Uses `aws-sigv4` directly, not
  the full AWS SDK.
- **S3c GCP IAM** — service account impersonation via JWT signing + token exchange. Reuses
  ed25519-dalek + reqwest already in tree.
- **S3d Kubernetes** — projected ServiceAccount tokens for the configured SA + audience.

### S4. Vaults + collections + items + folders (E2E, human surface)

Bitwarden-aligned data model.

**Per-user vault** (auto-created on signup, exactly one per user):

```
GET  /api/v1/vault/me                        # your personal vault
PUT  /api/v1/vault/me/items                  # add to your vault
GET  /api/v1/vault/me/items/{id}             # fetch ciphertext
```

**Collections** (shared, many per org, member-roleable):

```
POST /api/v1/vault/collections                              # create
POST /api/v1/vault/collections/{id}/members                 # add member with role
POST /api/v1/vault/collections/{id}/items                   # add item
GET  /api/v1/vault/collections/{id}/items/{id}              # fetch ciphertext
DELETE /api/v1/vault/collections/{id}/members/{user_id}     # revoke
```

**Folders** (Bitwarden-compat visual organization, not access):

```
POST /api/v1/vault/folders                  # personal folder
POST /api/v1/vault/collections/{id}/folders # collection folder
```

E2E encryption: items are encrypted with a per-vault or per-collection symmetric key. That key is
wrapped to each member's X25519 public key (envelope encryption). Server stores ciphertext +
envelopes; cannot decrypt either.

Membership is Zanzibar tuples (see "corporate hierarchy" section below for the full schema and
example).

### S5. Biscuit-share (assay-distinct)

Mint short-lived capability tokens for an item, vault, or collection. biscuit-attenuated:
time-bound, IP-bound, single-use. Verifies offline. Server-side revocation via `vault.share_revoked`
(by biscuit `key_id`).

```
POST /api/v1/vault/share        # mint token with caveats
GET  /vault/share/{token}       # public — returns item if caveats pass
POST /api/v1/vault/share/revoke # revoke a key_id
```

### S6. Bitwarden-protocol compat shim

Implements the subset of Bitwarden's HTTP API that lets stock Bitwarden mobile / browser / CLI
clients work against assay-engine.

In-compat scope for v0.3.0: `/api/sync`, `/api/ciphers`, `/api/folders`,
`/api/identity/connect/token`, passkey-as-cipher items, two-step auth.

Reference: `vaultwarden` at `/opt/forks/vaultwarden` (MIT — adaptable code).

Deferred-with-alternative (see deferred section below):

- BW Send → biscuit-share is the assay-native equivalent
- BW Organizations → collections + Zanzibar `org:` tuples
- BW Emergency Access → recovery group pattern
- BW file attachments → S3 + biscuit-share URLs

### S7. Sealing (KEK protection, ops-grade)

The master KEK protects every wrapped DEK in S1/S2 and the wrapped collection keys in S4. Ship three
sealing options out of the box:

- **Shamir Secret Sharing init unseal** — split the KEK into N shares; require K to unseal on boot.
  Pure-Rust crypto via `sharks` crate. Default for new installs without a cloud KMS.
- **Cloud KMS auto-unseal** — call AWS KMS `Encrypt` / `Decrypt` or GCP KMS
  `cryptoKeyVersions.encrypt` / `decrypt` to wrap/unwrap the KEK. Reuses `aws-sigv4` and JWT signing
  already in tree. Recommended for cloud deployments.
- **HSM via PKCS#11** — opt-in via `vault-sealing-hsm` feature flag. Uses `cryptoki` to talk to a
  PKCS#11 device for KEK unwrap. Default off — most operators don't have an HSM.

KEK rotation is a separate explicit operation (re-wrap every DEK); not automatic. Stored unseal
metadata in `vault.kek_metadata`.

### S8. Audit forwarding

In addition to the existing PG-backed audit table, forward events to external sinks:

- **syslog** — TCP / UDP / UDS. Native implementation, ~50 KB.
- **S3** — JSONL append per minute, configurable bucket/prefix. Reuses sigv4.
- **Webhook** — POST per event (or batched per N seconds) to a configured URL. Reuses reqwest.

Configurable per-sink filtering (e.g. forward only `vault.*` events to syslog, forward `auth.login`
to a separate webhook).

### S9. HA / failover

Already mostly there via `engine.instances` heartbeat + lease. Tighten the leader-lease so secondary
engine pods can take over within 10s of a primary failure. Document the failover playbook
explicitly. Postgres hot-follower replicas remain the caller's responsibility.

### S10. Dashboard

Mounted at `/vault/console`:

- KV browser (ops surface)
- Transit keys
- Dynamic leases
- My vault (personal items, folders)
- Collections (browse, create, members, items)
- Share manager (biscuit links, revoke)
- Sealing status (KEK kid, sealed/unsealed, unseal-shares-needed)
- Audit log

The **primary human UX is Bitwarden mobile/browser/CLI** via the S6 compat shim. Dashboard panes are
the admin overlay.

### S11. Lua stdlib — `assay.vault.*`

```lua
local vault = require("assay.vault")
local c = vault.client({ engine_url = "..." })

-- KV (ops)
c.kv:get("api/stripe")
c.kv:put("api/stripe", "...")

-- Transit
c.transit:encrypt("logs", "...")
c.transit:decrypt("logs", ct)

-- Dynamic creds
c.dynamic.aws:lease("ci-deploy", { ttl_secs = 1800 })
c.dynamic.postgres:lease("readonly")

-- Personal vault
c.me:add_item({ type = "password", ... })
c.me.folders:create("API keys")

-- Collections (shared)
local col = c.collections:create({ name = "Engineering" })
c.collections:add_member(col.id, "alice", "editor")
c.collections:add_item(col.id, { type = "password", ... })

-- Share
c.share:mint({ item_id = "...", ttl_secs = 3600, max_ip = "10.0.0.0/8" })
```

## Corporate hierarchy — the OpenBao / Vault equivalent

Both ops secrets (KV path tree) and human secrets (collections) support arbitrary corporate
hierarchies via Zanzibar tuples on prefixes. Concrete example for an org with engineering /
marketing / HR teams:

```
Acme Corp (Zanzibar object: org:acme)

# Ops secrets — KV path tree (server-decryptable)
kv_path:engineering/prod          ← admins: ops-leads team
kv_path:engineering/staging       ← writers: engineering team
kv_path:marketing/campaigns       ← admins: marketing team
kv_path:shared/sso-creds          ← writers: ops team

# Human secrets — collections (E2E)
collection:eng-onboarding         ← members: all engineers
collection:eng-prod-runbooks      ← members: senior engineers + on-call
collection:exec-ssh-keys          ← members: 4 named individuals
collection:hr-personnel-docs      ← members: HR team

# OIDC groups (from Cloudflare Access / Google Workspace)
group:engineering          → tuple: team:engineering#member
group:engineering-prod     → tuple: team:eng-prod-admins#member
group:senior-engineers     → tuple: team:senior-eng#member
group:marketing            → tuple: team:marketing#member
group:hr                   → tuple: team:hr#member

# Bindings (corporate admin sets these up)
kv_path:engineering/prod#admin     @ team:eng-prod-admins#member
kv_path:engineering/staging#writer @ team:engineering#member
kv_path:marketing/campaigns#admin  @ team:marketing#member
collection:eng-prod-runbooks#editor @ team:senior-eng#member
collection:hr-personnel-docs#admin  @ team:hr#member
```

A new hire joining Engineering gets `group:engineering` from OIDC on first sign-in → automatic
`team:engineering#member` tuple → automatic write access to `kv_path:engineering/staging` and read
on inherited paths. Promotion to senior engineer adds `team:senior-eng#member` → unlocks the
runbooks collection (an existing senior-eng member's client wraps the collection key for them on
next online presence). Departure: revoke OIDC tokens, drop the team tuples, the next-load forces
re-encryption envelopes for collections they belonged to.

Path hierarchy depth is unlimited; access tuples on a prefix cover all child paths. Same expressive
power as Vault's HCL `path "..."` ACLs, expressed as tuples.

### Engine wiring

- New module entry in `engine.modules`. Module name: `vault`.
- Cargo features (default-on group `vault`): `vault-kv`, `vault-transit`, `vault-collections`,
  `vault-share`, `vault-bitwarden-compat`, `vault-sealing-shamir`, `vault-sealing-kms`,
  `vault-audit-forwarding`, `vault-dynamic-postgres`, `vault-dynamic-aws`, `vault-dynamic-gcp`,
  `vault-dynamic-kubernetes`.
- Opt-in features (off by default): `vault-sealing-hsm` (PKCS#11).
- Schema namespace: `vault` (PG schema; SQLite attached file).
- `VaultCtx` composed into the central state struct, reuses the same backend pool the rest of the
  engine uses.

### Schema — tables in `vault.*`

- `vault.kv_meta` — KV path metadata (latest version, custom md, deletion state)
- `vault.kv` — versioned KV blobs (ciphertext, wrapped DEK, kek_kid)
- `vault.transit_keys` — name + algo + latest version
- `vault.transit_versions` — per-key version material (rotated)
- `vault.leases` — dynamic-creds lease tracking
- `vault.vaults` — per-user personal vault registry
- `vault.collections` — shared collections (org-scoped)
- `vault.collection_members` — (collection, user) → wrapped collection key + role
- `vault.items` — encrypted items (`vault_id` XOR `collection_id` set, `folder_id` optional)
- `vault.folders` — Bitwarden-compat folders (vault- or collection-scoped)
- `vault.share_revoked` — biscuit `key_id`s revoked
- `vault.kek_metadata` — sealing method, kid, unseal state
- `vault.unseal_shares` — Shamir SSS shares for KEK init unseal (encrypted-at-rest with the
  share-holder's key)
- `vault.audit_sinks` — forwarding configs (syslog/S3/webhook with filters)

### Crypto choices (locked)

| Concern                                  | Primitive                                                         | Why                                                |
| ---------------------------------------- | ----------------------------------------------------------------- | -------------------------------------------------- |
| KV at-rest encryption                    | AES-256-GCM-SIV (per-record DEK)                                  | Misuse-resistant AEAD. RustCrypto.                 |
| Transit                                  | AES-256-GCM-SIV with 96-bit nonce, version in AAD                 | Same primitive; nonce derived from version+counter |
| Collection / vault item encryption (E2E) | AES-256-GCM-SIV with collection key                               | Server cannot decrypt; client-side                 |
| Per-member envelope wrapping             | X25519 ECDH                                                       | Standard 1pw/Bitwarden shape                       |
| User keypair                             | X25519 (encryption) + Ed25519 (signing)                           | Already in tree via webauthn-rs / assay-auth       |
| Passkey-derived vault unlock             | HKDF over session-id + per-user salt + WebAuthn-attested material | No new flow — reuses session                       |
| Capability tokens (biscuit-share)        | biscuit (Datalog caveats)                                         | Already in tree, foundational in auth              |
| Master KEK                               | 256-bit symmetric, sealing-method-dependent at rest               | Shamir SSS / Cloud KMS / HSM (opt-in)              |
| Bitwarden-compat client KDF              | Argon2id (preferred) or PBKDF2-SHA256 (compat)                    | Both already in tree                               |
| Bitwarden-compat cipher encryption       | ChaCha20-Poly1305 (BW protocol)                                   | RustCrypto, ~50 KB                                 |
| Shamir Secret Sharing                    | `sharks` crate (Galois-field-256 SSS)                             | Pure Rust, audited                                 |

**No custom protocols. No homebrew crypto.** Every primitive has a peer-reviewed implementation in
the Rust ecosystem.

## Risk register

| Risk                                         | Likelihood | Severity | Mitigation                                                                                                                     |
| -------------------------------------------- | ---------- | -------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Custom crypto subtle bug                     | Med        | Critical | No custom protocols. Stick to ed25519-dalek + RustCrypto AEAD + audited `sharks` SSS. Crypto review pass before Phase 3 lands. |
| Bitwarden-compat scope creep                 | High       | Med      | Hard-freeze the supported endpoint list at start of Phase 7. Defer everything else.                                            |
| Schema lock-in                               | Med        | High     | Phase 0 schema lands separately from logic; let it run for a day before Phase 1+ commits to it.                                |
| Sealing edge cases (lost shares, KMS outage) | Med        | Critical | Document recovery playbook with each sealing method. Test the unseal-while-degraded path explicitly.                           |
| AWS / GCP minimal-SDK gaps                   | Med        | Low      | Two-call surface per provider; if we need more, fall back to fuller SDK. Behind feature flag either way.                       |
| Vault parity expectation                     | High       | Low      | Document the deferred-with-alternatives table prominently. Position as "Vault for the modern stack".                           |

## Open questions to resolve before Phase 1 commits

1. **Bitwarden client KDF default**. BW supports both PBKDF2 and Argon2id. Pick Argon2id for new
   accounts (matches assay-auth's password storage); accept PBKDF2 from imported BW vaults.
2. **Master KEK rotation cadence**. Reuse `auth.jwks_keys` rotation triggers, or independent?
   Probably independent — JWKS rotation is fast (issue new tokens), KEK rotation is slow (re-encrypt
   every DEK).
3. **biscuit issuer key separation**. Reuse the auth biscuit root key, or have a separate
   vault-biscuit-root? Separate keeps blast radius small if a key is compromised.
4. **AWS dynamic-creds first-customer integration**. The existing in-house project that connects to
   AWS — confirm exact endpoint shape before Phase 5 starts so the API matches their migration path.
5. **Cloud KMS key naming convention**. Auto-unseal config points at a specific KMS key; document
   the key-creation steps + IAM grants in `docs/migration-to-0.3.0.md`.

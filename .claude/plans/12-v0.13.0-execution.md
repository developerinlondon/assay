# 12 — v0.13.0 Execution Plan (control document)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans. Per-phase task detail lives in the sub-plans linked
> below; this document governs architecture, cross-cutting invariants, and phase gating.

**Goal:** Ship v0.13.0 as a single usable release with workflow + auth on a new engine architecture.
PostgreSQL 18 is the first-class backend; SQLite is the alternative for embedded / dev / test. Both
backends ship in the same `assay-engine` binary by default; the active backend is selected at
runtime from config. The `assay-lua` runtime stays small and scripting-focused; `assay-engine` is
the new HTTP server binary that hosts workflow, auth, and dashboard.

**Architecture in one sentence:** Each domain module (`assay-workflow`, `assay-auth`, future
`assay-vault` etc.) owns its own context type, router, and feature-gated backend impls;
`assay-engine` composes them via axum's `FromRef` so adding a module is "new crate + new field in
`EngineState`" with zero touching of existing modules.

**Tech stack:** Rust 2024, axum 0.8 (`FromRef` state composition), sqlx 0.8 (PG18 + SQLite),
`jsonwebtoken` 10 (`rust_crypto` feature), `openidconnect` 4, `oxide-auth` 0.6, `webauthn-rs` 0.5,
`argon2` + `password-hash` 0.5, `biscuit-auth` 6, tokio, utoipa, moon, cargo-workspaces,
testcontainers.

**Reference plans:** [10-assay-engine-architecture.md](./10-assay-engine-architecture.md) (target
architecture + diagrams), [11-engine-auth-modules.md](./11-engine-auth-modules.md) (auth module
details — consumed by Phases 4–7).

## Revision log

### v0.13.0 rev 3 — 2026-04-22 — split the release: engine-only now, auth in v0.14.0

Plan 12's original v0.13.0 goal bundled workflow and auth into a single release. After the rev-2
SurrealDB drop + Phase 3 engine-binary wiring shipped cleanly and tested against a running binary,
the conclusion was that the engine-split + workflow half was already a meaningful, demoable release
on its own. Holding it back behind Phases 4–7 (auth primitives, OIDC client + provider, passkey,
Zanzibar) adds roughly 50 hours of scope to a release that can otherwise go out the door.

The scope therefore splits in two. **v0.13.0** ships engine-split + PG18 + SurrealDB drop +
`assay-lua` simplification as its own release, with `assay-auth` present only as a scaffold crate.
**v0.14.0** picks up with Phase 4 (auth primitives) and covers everything up to Phase 8 (compose
auth into engine) plus Phases 9–10 (CI + docs + tag). Jeebon-V3 can start integrating against
v0.13.0 for workflow plumbing immediately; the IdP contract crystallises over v0.14.0 and jeebon
adopts it when it lands.

The v0.13.0 engine runs open (`AuthMode::no_auth()`) — any deployment that exposes it publicly needs
a network-layer gatekeeper (Cloudflare Access, Tailscale, VPN). v0.14.0 flips the default
workflow-API AuthMode to JWT-validating against the engine's own OIDC provider and gives the
dashboard auth views. Semver-wise, 0.13.0 → 0.14.0 is additive for HTTP consumers — existing
workflow endpoints keep their shape — but library consumers of `assay-auth` will see the crate grow
from a scaffold to its full surface.

Per-crate release versions at v0.13.0 ship time: `assay-lua` 0.13.0, `assay-workflow` 0.2.0
(breaking vs 0.1.x), `assay-core` / `assay-dashboard` / `assay-engine` / `assay-auth` all 0.1.0 as
new crates.

### v0.13.0 rev 2 — 2026-04-22 — drop SurrealDB, target PG18

**Decision:** SurrealDB is removed from v0.13.0 entirely. Backends are PostgreSQL 18 (default) and
SQLite (alternative). Both compile into the `assay-engine` binary by default; the active backend is
picked at startup based on the `EngineConfig.backend` discriminant.

**Why (evidence, not speculation):**

1. **Compile cost** (measured on `crates/assay-lua` clean release build on the project workspace):

   | Metric               | PG + SQLite | + SurrealDB | Delta    |
   | -------------------- | ----------- | ----------- | -------- |
   | Wall clock           | 91 s        | 281 s       | **3.1×** |
   | Peak RAM             | 1.28 GB     | 3.67 GB     | **2.9×** |
   | Stripped binary size | 14.0 MB     | 14.8 MB     | +5.5%    |

   Binary size is negligible; build time and RAM are not. `surrealdb-core` is pulled transitively
   even when only the remote-client features are enabled (`protocol-ws + protocol-http + rustls`) —
   there is no thin-client variant in the 3.x crate.

2. **Crypto-stack conflict was a myth.** Earlier plan text claimed `surrealdb`'s transitive
   `jsonwebtoken` + `aws-lc-rs` feature-unified with `rust_crypto` in `assay-lua`.
   `cargo tree -p assay-workflow --no-default-features --features "backend-postgres backend-sqlite"`
   shows `jsonwebtoken 10.3.0`, `aws-lc-rs 1.15`, `ring 0.17`, and `rustls 0.23` already in the
   baseline build (pulled by `sqlx` + `reqwest`). Surreal wasn't the source.

3. **No lost capability.** PostgreSQL 18 + `pgvector` + recursive CTEs covers the Zanzibar tuple
   store, document hierarchies, family-tree traversal, and vector search that SurrealDB marketing
   promised. Graph-shaped product workloads with bounded depth (≤5 hops) run in single-digit
   milliseconds on a PG18 tuple table of millions of rows with the new skip-scan composite index.

4. **Ecosystem maturity.** PG18 is GA on AWS RDS, Google Cloud SQL, Azure Flex Server, Neon
   (preview), and self-hosts trivially. SurrealDB 3.x has a far smaller operator ecosystem.

**What it changes:**

- `assay-workflow`, `assay-auth`: the `backend-surrealdb` feature + `src/store/surrealdb.rs` impls
  are removed entirely. `WorkflowStore` / `ZanzibarStore` / `UserStore` / `SessionStore` traits in
  `assay-core` stay as-is (backend-swappable) — only the backend count drops from 3 to 2.
- `assay-engine`: the `backend-surrealdb` feature is removed. `Backend::Surreal` variant is removed
  from the runtime config enum. `Backend::Postgres` and `Backend::Sqlite` remain and both ship in
  the default binary.
- Phase 3 of plan 12 (SurrealDB workflow backend, 12 hours, 16 store methods) is **deleted**. The
  trait-abstraction lesson from that phase is retained; the impl code is removed with the branch.
- Phase 6 of plan 12c (Zanzibar core) keeps the `ZanzibarStore` trait but implements PG + SQLite
  only. The "Zanzibar SurrealDB impl" section in plan 11 is struck.
- CI matrix: drops SurrealDB service container from GitHub Actions. No SurrealDB testcontainer.

**Backend feature model — additive, not exclusive:**

```toml
# assay-workflow/Cargo.toml
[features]
default = ["backend-postgres", "backend-sqlite"]
backend-postgres = ["dep:sqlx", "sqlx/postgres", "sqlx/runtime-tokio-rustls", "sqlx/macros"]
backend-sqlite = ["dep:sqlx", "sqlx/sqlite", "sqlx/runtime-tokio-rustls", "sqlx/macros"]
```

Both features can be enabled together (and are, by default). Each crate compiles both
`src/store/postgres.rs` and `src/store/sqlite.rs` when both features are on. The engine selects one
at startup:

```rust
// crates/assay-engine/src/main.rs
match cfg.backend {
    Backend::Postgres { url } => run_engine::<PostgresStore>(cfg).await,
    Backend::Sqlite   { path } => run_engine::<SqliteStore>(cfg).await,
}
```

Users who want a minimal binary (embedded SQLite-only, no PG driver linked) can build with
`--no-default-features --features backend-sqlite`. Users on PG-only infrastructure can build with
`--no-default-features --features backend-postgres`. The default includes both because the
compile-size delta is a few MB and it lets a single distributed binary serve multiple deployment
shapes.

## PostgreSQL 18 features to leverage

PG18 landed 2025-09-25. For v0.13 we target **PG18 minimum**. The features we build on:

| Feature                        | Where we use it                                                                                                                                                                                            |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `uuidv7()` built-in            | Primary keys for: `workflows.id`, `events.id`, `activities.id`, `timers.id`, `zanzibar_tuple.id`, `users.id`, `sessions.id`. Time-ordered, B-tree friendly, no app-side UUID gen needed.                   |
| Skip-scan on B-tree            | Zanzibar tuple lookups. One composite index `(object_type, object_id, relation, subject_type, subject_id)` now serves forward + inverse queries with different leading-column filters. No partial indexes. |
| Asynchronous I/O (`io_uring`)  | Workflow task-polling scans (`SELECT … FOR UPDATE SKIP LOCKED`) benefit from AIO on large queue tables. `io_method = io_uring` recommended for self-host on Linux kernels ≥ 5.13.                          |
| Virtual generated columns      | Denormalised lookup keys on JSONB event payloads — e.g. `workflow_id_gen` extracted from `payload->>'workflow_id'` for indexing without store duplication.                                                 |
| `jsonb_strip_nulls(..., true)` | Workflow event payload normalisation — strip null array elements before indexing.                                                                                                                          |
| OAuth in libpq (optional)      | Not used for v0.13. Noted for future: if we add DB-level OIDC identity passthrough, this is the hook.                                                                                                      |

**SQLite equivalent notes:** SQLite gets app-generated UUIDv7 (the `uuid` crate's `Uuid::now_v7()`),
single-column indexes per query pattern (no skip-scan), and standard synchronous I/O. Identical
`WorkflowStore` / `ZanzibarStore` semantics; feature parity, just different internals per driver.

**Not yet available, do not plan around:**

- **SQL/PGQ (ISO SQL property-graph queries)** — committed to PostgreSQL 19 (2026-03-16,
  [commitfest #4904](https://commitfest.postgresql.org/patch/4904/)), releases ≈ Sept/Oct 2026. PG18
  uses recursive CTEs. Migration to PGQ syntax on PG19+ is additive — schemas don't change.
- **Apache AGE** — not adopted. Recursive CTEs cover our bounded-depth graph queries; AGE would add
  `shared_preload_libraries` + superuser install friction for no gain at this depth.

## Architecture principles (non-negotiable)

### 1. `FromRef` state composition — one state type per module

Each module crate owns a plain-old-struct context type and a `fn router() -> Router<Ctx>`. The
engine composes all module contexts into one `EngineState` and derives axum's `FromRef` so each
sub-router receives its slice without knowing about the others.

```rust
// in assay-workflow
pub struct WorkflowCtx<S: WorkflowStore> {
    pub(crate) store: Arc<S>,
    pub event_tx: broadcast::Sender<BroadcastEvent>,
    pub(crate) _bg: Arc<BackgroundTasks>,  // scheduler + timer + health handles
    pub auth_mode: AuthMode,
    pub binary_version: Option<&'static str>,
}
impl<S: WorkflowStore> WorkflowCtx<S> {
    pub fn start(store: S) -> Self { /* spawns background tasks */ }
    pub async fn start_workflow(&self, ...) -> Result<...> { ... }
    // ... all the orchestration methods live here (split across files by concern)
}
pub fn router<S: WorkflowStore + 'static>() -> Router<Arc<WorkflowCtx<S>>> { ... }

// in assay-auth (no generic — auth doesn't have the RPITIT problem)
pub struct AuthCtx {
    pub users: Arc<dyn UserStore>,
    pub sessions: Arc<dyn SessionStore>,
    pub zanzibar: Arc<dyn ZanzibarStore>,
    pub oidc: Arc<OidcProvider>,
    pub jwt: JwtConfig,
}
pub fn router() -> Router<Arc<AuthCtx>> { ... }

// in assay-engine — EngineState generic on S (backend selected at startup)
#[derive(Clone)]
pub struct EngineState<S: WorkflowStore> {
    pub workflow: Arc<WorkflowCtx<S>>,
    pub auth: Arc<AuthCtx>,
    pub dashboard: Arc<DashboardCtx>,
}
// FromRef impls for each sub-ctx (derive or hand-written)
impl<S: WorkflowStore> FromRef<EngineState<S>> for Arc<WorkflowCtx<S>> {
    fn from_ref(s: &EngineState<S>) -> Self { Arc::clone(&s.workflow) }
}
impl<S: WorkflowStore> FromRef<EngineState<S>> for Arc<AuthCtx> { ... }
impl<S: WorkflowStore> FromRef<EngineState<S>> for Arc<DashboardCtx> { ... }

// engine main() picks the concrete S based on runtime config:
match cfg.backend {
    Backend::Postgres { url } => run_engine::<PostgresStore>(cfg).await,
    Backend::Sqlite   { path } => run_engine::<SqliteStore>(cfg).await,
}

async fn run_engine<S: WorkflowStore + 'static>(cfg: EngineConfig) -> Result<()> {
    let state = EngineState::<S>::build(&cfg).await?;
    let app = Router::new()
        .merge(assay_workflow::router::<S>())
        .merge(assay_auth::router())
        .merge(assay_dashboard::router())
        .with_state(state);
    serve(app, cfg.server.bind_addr).await
}
```

**Rules:**

- Modules NEVER import each other's context types. Cross-module calls go through shared traits in
  `assay-core`, or through the engine's wiring code (which is fine — wiring is the engine's job).
- Handlers use `State<Arc<WorkflowCtx<S>>>`, `State<Arc<AuthCtx>>`, etc. — `FromRef` extracts the
  per-module `Arc<Ctx>` from `EngineState<S>`. One atomic refcount bump per request.
- Workflow handlers are generic over `S: WorkflowStore` (the trait's `impl Future` returns prevent
  dyn). Auth/dashboard handlers are not generic. Engine picks `S` at startup via `match cfg.backend`
  and runs one monomorphisation per backend compiled in.
- **One struct per module owns both orchestration AND axum state.** `WorkflowCtx` holds the store,
  broadcaster, and background-task handles; it IS the workflow orchestrator. There is no separate
  `Engine` struct.

### 2. Generic `<S: Store>` cascade kept — monomorphised static dispatch

> **Revision (2026-04-21):** The original text proposed `Arc<dyn WorkflowStore>` trait-object
> dispatch. Not possible — `WorkflowStore` has `impl Future` return types (RPITIT), which make it
> non-dyn-compatible on current Rust. Rather than rewrite the trait to `Box<dyn Future>` returns
> (heap allocation per call — hot in workflow replay where we read hundreds of events per request),
> we **keep the generic parameter**.

**What the codebase looks like:**

- `pub struct WorkflowCtx<S: WorkflowStore>` holds `Arc<S>`. Methods are monomorphised per backend.
- Handlers in `assay-workflow::api` have `<S: WorkflowStore>` in their signatures.
- Only the `assay-engine` binary embeds `WorkflowCtx<S>`. It compiles one monomorphisation per
  backend feature it has enabled. Backend selection happens at startup via `match cfg.backend`.
- The `assay-lua` runtime does NOT embed a workflow engine or backend — it is a pure Lua runtime
  that talks to a deployed `assay-engine` over HTTP (see Principle 8).

**Cost:** each backend that's compiled-in adds one monomorphisation of workflow code (~20–40 KB of
text per backend in a release build — noise vs the 15–30 MB engine binary). Zero runtime cost
(inlined static dispatch, no allocations, no vtable lookups). With both PG and SQLite features
enabled (the default), fan-out is 2× — measured impact is negligible.

**Why not the boxed-future alternative:**

- Workflow replay calls `store.list_events(id).await` then iterates hundreds of event rows per
  deterministic replay. Each call boxing its future = ~hundreds of heap allocations per request.
- The generic keeps the hot path allocation-free.
- Cost-benefit: simpler dispatch model (generic) vs. simpler ergonomics (dyn). At our trait size
  (~40 methods) the generic is manageable.

**Where this shows up in Phase 8 (`assay-engine`):**

Since `WorkflowCtx<S>` is generic, `EngineState<S>` is also generic:

```rust
#[derive(Clone)]
pub struct EngineState<S: WorkflowStore> {
    pub workflow: Arc<WorkflowCtx<S>>,
    pub auth: Arc<AuthCtx>,         // auth isn't generic — no store generic cascade there
    pub dashboard: Arc<DashboardCtx>,
}
```

The engine binary's `main()` picks a concrete `S` based on config:

```rust
match cfg.backend {
    #[cfg(feature = "backend-postgres")]
    Backend::Postgres { url } => run_engine::<PostgresStore>(cfg).await,
    #[cfg(feature = "backend-sqlite")]
    Backend::Sqlite   { path } => run_engine::<SqliteStore>(cfg).await,
}
```

Each enabled backend compiles as a separate `run_engine` monomorphisation. Server routing + handlers
live inside `run_engine<S>`, parametric on the chosen backend. Runtime overhead of the `match` is a
single branch at startup, never on the hot path.

**Future:** if Rust stabilises dyn-compatible trait methods with `impl Future` returns, or if the
project's hot paths change character, we can revisit. Not a 0.13.0 concern.

### 3. Layout 1 — backends feature-gated inside the module crate

```
crates/assay-workflow/src/store/
├── mod.rs
├── postgres.rs     #[cfg(feature = "backend-postgres")]
└── sqlite.rs       #[cfg(feature = "backend-sqlite")]

crates/assay-auth/src/store/
├── mod.rs
├── postgres.rs     #[cfg(feature = "backend-postgres")]
└── sqlite.rs       #[cfg(feature = "backend-sqlite")]
```

**Why not one crate per backend:** trait churn (workflow store grows a method) would require
coordinated releases across N backend crates. Layout 1 keeps module evolution local.

**Features are additive** (not mutually exclusive): a single binary can ship with both PG and SQLite
support, letting one binary serve multiple deployment shapes. Operators pick at startup via the
`backend` config field.

**Adding a new backend** (e.g. MySQL, CockroachDB): add `src/store/<name>.rs` under each module
crate, add `backend-<name>` feature, extend the `Backend` enum, done.

**Adding a new module** (e.g. vault): new crate `assay-vault` with its own
`src/store/{postgres,sqlite}.rs`, wire into `EngineState` in `assay-engine`, done.

### 4. Shared connection pool at the engine level

When workflow + auth both talk to Postgres, one pool serves both. The engine constructs the pool;
each module accepts it via `from_pool(pool: PgPool) -> Self`.

```rust
// assay-engine main.rs, Postgres branch
let pg = PgPoolOptions::new()
    .max_connections(cfg.pg.max_connections)
    .connect(&cfg.pg_url)
    .await?;
let workflow_ctx = WorkflowCtx::from_pg_pool(pg.clone());
let auth_ctx = AuthCtx::from_pg_pool(pg.clone());
```

Same for SQLite — engine opens one `SqlitePool`, clones to each module. Each module's store impl is
a thin wrapper that runs queries through the shared connection.

### 5. Per-module migrations, applied in order by the engine

Each module owns its own `migrations/` tree and its own migrations tracker table. The engine's
startup applies them in a deterministic order, in the branch matching the active backend:

```rust
match cfg.backend {
    Backend::Postgres { .. } => {
        assay_workflow::migrate::postgres(&pg).await?;
        assay_auth::migrate::postgres(&pg).await?;
    }
    Backend::Sqlite { .. } => {
        assay_workflow::migrate::sqlite(&sqlite).await?;
        assay_auth::migrate::sqlite(&sqlite).await?;
    }
}
```

Tracker tables are namespaced: `_assay_workflow_migrations`, `_assay_auth_migrations`. They don't
collide even in the shared-pool case.

### 6. Parametrised integration testing across both backends

Every store-level test runs against PG18 and SQLite via a `Harness` enum (rstest-driven). A test
that passes on one backend but fails on the other is a leaky contract — fix the trait or the
backend, never paper over it. Both backends compile together by default, so a single
`cargo test --workspace` run exercises both.

### 7. PostgreSQL 18 is the minimum supported version

- `assay-workflow` and `assay-auth` migrations target PG 18.x syntax and features (notably
  `uuidv7()` and skip-scan-friendly composite indexes).
- Self-hosted deployments: `postgres:18-alpine` Docker image (~245 MB), `pgvector/pgvector:pg18`
  variant (~156 MB compressed) if jeebon or another consumer needs vector search in the same DB.
- Managed: AWS RDS PG18, Google Cloud SQL PG18, Azure Flexible Server PG18, Neon PG18 (preview),
  Fly.io `postgres-flex`. Supabase's managed PG18 is not yet available (PG17 default) — consumers on
  Supabase must self-host or wait.
- CI: GitHub Actions `services.postgres: postgres:18-alpine`.

### 8. `assay-lua` is a pure Lua runtime — no embedded workflow, no auth

`assay-lua` becomes a thin binary:

- Lua 5.5 via `mlua` + stdlib (fs, string, json, http client, etc.).
- HTTP client wrappers that call an `assay-engine` instance over HTTP (for workflow, auth, Zanzibar
  checks). `ASSAY_ENGINE_URL` env var or `--engine` CLI flag points to the engine.
- No `assay-workflow` dep. No `assay-auth` dep. No `sqlx` dep. No embedded DB. No server.

**Why:** `assay-engine` is the single source of truth for workflow state and identity. Embedding a
second copy inside `assay-lua` doubled the build, forked the trait abstractions, and created two
places where bugs could live. Scripts that need workflow or auth run against a deployed engine;
scripts that don't need them (data munging, automation, CI glue) pay zero auth/workflow cost.

**What this looks like in Lua:**

```lua
-- in a script
local engine = assay.engine("http://localhost:3000")
local wf = engine:start_workflow("my-queue", { payload = { ... } })
local can = engine:auth_check("doc:123", "view", "user:alice")
```

Under the hood: HTTP calls, JWT auth to the engine, retries, streaming via SSE for long-running ops.
No process co-location required — script and engine can live on different hosts.

**Build surface for `assay-lua`**: `mlua`, `reqwest`, `tokio`, `serde`, `serde_json`, `clap`,
`jsonwebtoken` (for signing requests to the engine). That's it. Binary target: ≤ 10 MB stripped
release on Linux x86_64 musl.

## Backend parity matrix — what works on PG18 vs SQLite

Both backends implement the full `WorkflowStore` / `ZanzibarStore` / `UserStore` / `SessionStore`
trait surface. Functional parity is enforced by the parametrised test harness (principle 6).
Internals differ:

| Capability                    | PG18                              | SQLite                                                             |
| ----------------------------- | --------------------------------- | ------------------------------------------------------------------ |
| Time-ordered UUID PK          | `DEFAULT uuidv7()` built-in       | App-side `uuid::Uuid::now_v7()`                                    |
| Zanzibar tuple index strategy | One composite (PG18 skip-scan)    | Two composites — forward `(obj,rel,sub)` + reverse `(sub,rel,obj)` |
| Recursive CTE (userset walk)  | `WITH RECURSIVE`                  | `WITH RECURSIVE` (same)                                            |
| Zookie consistency token      | `pg_snapshot_xmin` + MVCC snap.   | Monotonic `revision` counter in `zanzibar_zookie` table            |
| Concurrency                   | Many writers + many readers       | One writer + many readers (WAL mode)                               |
| Workflow queue polling        | `SELECT … FOR UPDATE SKIP LOCKED` | `UPDATE … RETURNING` atomic claim                                  |
| Workflow push streams         | `LISTEN/NOTIFY` real-time         | **Polling only** — no in-DB pub/sub                                |
| JSON payloads                 | Native `JSONB`                    | `TEXT` + JSON1 extension (built-in since 3.38)                     |
| Vector search (`pgvector`)    | HNSW indexes                      | Not native — offload to `sqlite-vec` or external service           |
| Asynchronous I/O              | `io_uring` (PG18)                 | Standard synchronous                                               |
| Full-text search              | `tsvector` + GIN                  | FTS5 extension                                                     |

**When to pick which:**

- **PG18** — multi-tenant SaaS, multi-process writers, real-time push, vector search, millions of
  tuples. Default for any production deployment. Required for jeebon.
- **SQLite** — embedded, single-process, dev / test / CI, personal self-host, offline-first apps.
  All auth modules (OIDC, passkey, JWT, Biscuit, session, Zanzibar) work fully. No vector search.

**Invariant:** a test that passes on one backend but fails on the other is a leaky contract — fix
the trait or the backend impl, never paper over it in handler code.

## Cross-cutting invariants

### Error types

Every crate defines its own error enum (`assay_core::Error`, `assay_workflow::Error`,
`assay_auth::Error`). `anyhow::Result` is used at the handler boundary only. `thiserror` for
definitions.

`assay-core::Error` is the common parent for types that cross crate boundaries. Backend errors
(sqlx) convert INTO the module's error, never leak out as their native type.

### Logging / tracing

Every public function gets `#[tracing::instrument(skip(self, ...))]` when it returns a non-trivial
error. Log levels: `error!` for actionable failures, `warn!` for recovered failures, `info!` for
lifecycle events, `debug!` for high-volume internal state.

Never log: passwords, session cookies, JWT signatures, Biscuit tokens, OIDC tokens, Zanzibar
relation tuples (they encode permission topology). The tracing filter must redact or skip these.

### Testing strategy

- **Unit tests**: in-crate, `#[cfg(test)]` modules, no DB, no network.
- **Integration tests**: `crates/<crate>/tests/*.rs`, one file per feature area, parametrised over
  backend via `rstest` + `Harness`.
- **End-to-end tests**: `crates/assay-engine/tests/e2e/`, start the engine binary in a subprocess,
  hit it with `reqwest`.
- **Dashboard e2e**: existing Playwright suite at `crates/assay-workflow/tests-e2e/` — move to
  `crates/assay-dashboard/tests-e2e/` in Phase 8.

### TDD discipline

Per-task format in every sub-plan follows the writing-plans skill:

1. Write the failing test
2. Run it, observe the failure
3. Implement the minimum to pass
4. Run the test, observe it passes
5. Commit

The sub-plans may collapse trivial mechanical steps (`git mv`, `cargo check`) into one step when TDD
doesn't apply.

### Dependency versions

Version numbers cited in sub-plans are **minimum known-good** as of the plan's writing (2026-04-22).
When a sub-plan adds a dependency to a `Cargo.toml`, the subagent MUST:

1. Run `cargo add <crate>` (or manually check crates.io) to get the current stable major/minor.
2. If the latest major differs from what the plan cites, update the plan first, then proceed — don't
   silently diverge.
3. Record the resolved version in the commit message body (e.g. `resolved: openidconnect@4.0.1`).

Exceptions documented explicitly in plans:

- `argon2` + `password-hash` stay at **0.5** — argon2 0.6 is still RC as of 2026-04-22 (see plan 11
  dependency notes). Bump both together once argon2 0.6 ships stable.
- `jsonwebtoken` pinned to **10.x with `features = ["rust_crypto"]`** across all assay crates so
  Cargo feature unification settles on the pure-Rust back-end without pulling in duplicate crypto
  primitives.

### Commit granularity

One task = one commit. Commit messages follow the existing Conventional Commits style:

```
<type>(<scope>): <summary>

Optional body paragraphs explaining why, not what.
```

Types: `feat`, `fix`, `refactor`, `test`, `docs`, `ci`, `chore`. Scopes: crate name (`core`,
`workflow`, `auth`, `engine`, `dashboard`, `runtime`) or area (`workflow/pg`, `auth/zanzibar`,
`engine/config`).

## Global exit criteria for v0.13.0

Before tagging `v0.13.0`, all of these must hold:

1. **Build matrix:** every combination of `--features` listed in each crate's `Cargo.toml` compiles
   clean (`cargo check` per combination, not just defaults).
2. **Test matrix:** `cargo test --workspace` green. Parametrised backend tests pass on both PG and
   SQLite.
3. **No `unwrap()` in library code.** `grep -rn 'unwrap()' crates/*/src | grep -v test` returns only
   entries with a nearby `// SAFETY:` or `// ok: <reason>` comment.
4. **No `todo!()` / `unimplemented!()` outside test scaffolding.** Same grep discipline.
5. **Runtime smoke test** — `assay run crates/assay/tests/fixtures/hello.lua` works and matches
   pre-split output.
6. **Engine smoke test** — `assay-engine --config examples/sqlite.toml` serves `/healthz`,
   `/api/v1/workflows`, `/authorize`, `/dashboard`, and each returns the expected shape. Same with
   `examples/postgres.toml` against a PG18 testcontainer.
7. **Zanzibar smoke test** — a script that creates namespace schema, writes tuples, and runs
   `check()` against both backends returns identical results.
8. **Size budget** — release binaries measured on Linux x86_64 musl, stripped:
   - `assay-lua` runtime (Lua + stdlib + HTTP client, no workflow, no auth): ≤ 10 MB (goal 6–10 MB).
   - `assay-engine` with auth + dashboard + workflow + both backends: ≤ 20 MB (goal 15–20 MB).
   - Fail the release if any is > 10% over goal.
9. **Docs** — CHANGELOG per changed crate, README updated with engine/runtime split, migration notes
   for pre-0.13 consumers, PG18 install guide, backend selection guide.
10. **Per-crate git tags pushed** — `assay-v0.13.0`, `assay-engine-v0.1.0`, `assay-workflow-v0.2.0`,
    `assay-core-v0.1.0`, `assay-dashboard-v0.1.0`, `assay-auth-v0.1.0`.

## Rollback strategy

This is a big refactor landing on `feature/0.13.0-engine-split`. Rollback options at each gate:

| Stage              | If we need to bail                                  | How                                                                                                          |
| ------------------ | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| Phase 0 done       | Keep scaffold commits; discard later work           | `git reset --hard <phase-0-end-sha>` on the branch                                                           |
| Phase 1 done       | State refactor intact; workflow/auth haven't landed | Ship 0.12.2 with workspace structure only, no auth                                                           |
| Rev-2 strip done   | PG+SQLite workflow ready; auth untouched            | Ship 0.12.3-preview as runtime+workflow release                                                              |
| Phases 4–7 partial | Auth incomplete, ship behind feature flag           | Mark `assay-auth` as experimental in README, remove from default engine features, ship runtime+workflow only |
| Full plan done     | —                                                   | Ship v0.13.0 as specified                                                                                    |

The branch stays open until 0.13.0 ships. No cherry-picking partial changes to `main` unless the
owning phase has passed its exit criteria.

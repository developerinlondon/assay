# 12 — v0.13.0 Execution Plan (control document)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans. Per-phase task detail lives in the sub-plans linked
> below; this document governs architecture, cross-cutting invariants, and phase gating.

**Goal:** Ship v0.13.0 as a single usable release with workflow + auth on a new engine architecture.
SurrealDB becomes a first-class backend alongside Postgres and SQLite. The `assay` Lua runtime stays
small and scripting-focused; `assay-engine` is the new HTTP server binary that hosts workflow, auth,
and dashboard on pluggable backends.

**Architecture in one sentence:** Each domain module (`assay-workflow`, `assay-auth`, future
`assay-vault` etc.) owns its own context type, router, and feature-gated backend impls;
`assay-engine` composes them via axum's `FromRef` so adding a module is "new crate + new field in
`EngineState`" with zero touching of existing modules.

**Tech stack:** Rust 2024, axum 0.8 (`FromRef` state composition), sqlx 0.8 (PG/SQLite), `surrealdb`
3.x (remote-only, `protocol-ws + protocol-http + rustls`), `jsonwebtoken` 10, `openidconnect` 4,
`oxide-auth` 0.6, `webauthn-rs` 0.5, `argon2` + `password-hash` 0.5, `biscuit-auth` 6, tokio,
utoipa, moon, cargo-workspaces, testcontainers.

**Reference plans:** [10-assay-engine-architecture.md](./10-assay-engine-architecture.md) (target
architecture + diagrams), [11-engine-auth-modules.md](./11-engine-auth-modules.md) (auth module
details — consumed by Phases 4–7).

## Architecture principles (non-negotiable)

### 1. `FromRef` state composition — one state type per module

Each module crate owns a plain-old-struct context type and a `fn router() -> Router<Ctx>`. The
engine composes all module contexts into one `EngineState` and derives axum's `FromRef` so each
sub-router receives its slice without knowing about the others.

```rust
// in assay-workflow
pub struct WorkflowCtx<S: WorkflowStore> {
    pub(crate) store: Arc<S>,
    pub event_tx: broadcast::Sender<BroadcastEvent>,
    pub(crate) _bg: Arc<BackgroundTasks>,  // scheduler + timer + health handles
    pub auth_mode: AuthMode,
    pub binary_version: Option<&'static str>,
}
impl<S: WorkflowStore> WorkflowCtx<S> {
    pub fn start(store: S) -> Self { /* spawns background tasks */ }
    pub async fn start_workflow(&self, ...) -> Result<...> { ... }
    // ... all the orchestration methods live here (split across files by concern)
}
pub fn router<S: WorkflowStore + 'static>() -> Router<Arc<WorkflowCtx<S>>> { ... }

// in assay-auth (no generic — auth doesn't have the RPITIT problem)
pub struct AuthCtx {
    pub users: Arc<dyn UserStore>,
    pub sessions: Arc<dyn SessionStore>,
    pub zanzibar: Arc<dyn ZanzibarStore>,
    pub oidc: Arc<OidcProvider>,
    pub jwt: JwtConfig,
}
pub fn router() -> Router<Arc<AuthCtx>> { ... }

// in assay-engine — EngineState generic on S (backend selected at startup)
#[derive(Clone)]
pub struct EngineState<S: WorkflowStore> {
    pub workflow: Arc<WorkflowCtx<S>>,
    pub auth: Arc<AuthCtx>,
    pub dashboard: Arc<DashboardCtx>,
}
// FromRef impls for each sub-ctx (derive or hand-written)
impl<S: WorkflowStore> FromRef<EngineState<S>> for Arc<WorkflowCtx<S>> {
    fn from_ref(s: &EngineState<S>) -> Self { Arc::clone(&s.workflow) }
}
impl<S: WorkflowStore> FromRef<EngineState<S>> for Arc<AuthCtx> { ... }
impl<S: WorkflowStore> FromRef<EngineState<S>> for Arc<DashboardCtx> { ... }

// engine main() picks the concrete S based on config:
match cfg.backend {
    Backend::Postgres { url } => run_engine::<PostgresStore>(cfg).await,
    Backend::Sqlite   { path } => run_engine::<SqliteStore>(cfg).await,
    Backend::Surreal  { .. }   => run_engine::<SurrealDbStore>(cfg).await,
}

async fn run_engine<S: WorkflowStore + 'static>(cfg: EngineConfig) -> Result<()> {
    let state = EngineState::<S>::build(&cfg).await?;
    let app = Router::new()
        .merge(assay_workflow::router::<S>())
        .merge(assay_auth::router())
        .merge(assay_dashboard::router())
        .with_state(state);
    serve(app, cfg.server.bind_addr).await
}
```

**Rules:**

- Modules NEVER import each other's context types. Cross-module calls go through shared traits in
  `assay-core`, or through the engine's wiring code (which is fine — wiring is the engine's job).
- Handlers use `State<Arc<WorkflowCtx<S>>>`, `State<Arc<AuthCtx>>`, etc. — `FromRef` extracts the
  per-module `Arc<Ctx>` from `EngineState<S>`. One atomic refcount bump per request.
- Workflow handlers are generic over `S: WorkflowStore` (the trait's `impl Future` returns prevent
  dyn). Auth/dashboard handlers are not generic. Engine picks `S` at startup based on config and
  runs a single monomorphisation. See principle 2 for rationale.
- **One struct per module owns both orchestration AND axum state.** `WorkflowCtx` holds the store,
  broadcaster, and background-task handles; it IS the workflow orchestrator. There is no separate
  `Engine` struct.

### 2. Generic `<S: Store>` cascade kept — monomorphised static dispatch

> **Revision (2026-04-21):** The original text here proposed `Arc<dyn WorkflowStore>` trait-object
> dispatch. That's not possible with the current trait: `WorkflowStore` has `impl Future` return
> types (RPITIT), which make it non-dyn-compatible on current Rust. Rather than rewrite the trait to
> `Box<dyn Future>` returns (which would add a heap allocation per call — hot in workflow replay
> where we read hundreds of events per request), we **keep the generic parameter**.

**What the codebase looks like:**

- `pub struct WorkflowCtx<S: WorkflowStore>` holds `Arc<S>`. Methods are monomorphised per backend.
- Handlers in `assay-workflow::api` have `<S: WorkflowStore>` in their signatures.
- Each binary (`assay` runtime, `assay-engine`) selects a concrete `S` at startup based on config
  and dispatches to the matching monomorphisation.

**Cost:** each backend that's compiled-in adds a separate monomorphisation of workflow code (~20-40
KB of text per backend in a release build — noise vs the 30 MB engine binary). Zero runtime cost
(inlined static dispatch, no allocations, no vtable lookups).

**Why not the boxed-future alternative:**

- Workflow replay calls `store.list_events(id).await` then iterates hundreds of event rows per
  deterministic replay. Each call boxing its future = ~hundreds of heap allocations per request.
- The generic keeps the hot path allocation-free.
- Cost-benefit: simpler dispatch model (generic) vs. simpler ergonomics (dyn). At our trait size
  (~40 methods) the generic is manageable.

**Where this shows up in Phase 8 (`assay-engine`):**

Since `WorkflowCtx<S>` is generic, `EngineState<S>` is also generic:

```rust
#[derive(Clone)]
pub struct EngineState<S: WorkflowStore> {
    pub workflow: Arc<WorkflowCtx<S>>,
    pub auth: Arc<AuthCtx>,         // auth isn't generic — no store generic cascade there
    pub dashboard: Arc<DashboardCtx>,
}
```

The engine binary's `main()` picks a concrete `S` based on config:

```rust
match cfg.backend {
    Backend::Postgres { url } => run_engine::<PostgresStore>(cfg).await,
    Backend::Sqlite   { path } => run_engine::<SqliteStore>(cfg).await,
    Backend::Surreal  { .. }   => run_engine::<SurrealDbStore>(cfg).await,
}
```

Each backend compiles as a separate `run_engine` monomorphisation. Server routing + handlers live
inside `run_engine<S>`, parametric on the chosen backend. Runtime overhead: zero.

**Future:** if Rust stabilises dyn-compatible trait methods with `impl Future` returns, or if the
project's hot paths change character, we can revisit. Not a 0.13.0 concern.

**The one exception:** the `Engine` struct itself, if it needs to own a concrete store type for
lifetime reasons, can stay generic — but the router layer doesn't.

### 3. Layout 1 — backends feature-gated inside the module crate

```
crates/assay-workflow/src/store/
├── mod.rs
├── postgres.rs     #[cfg(feature = "backend-postgres")]
├── sqlite.rs       #[cfg(feature = "backend-sqlite")]
└── surrealdb.rs    #[cfg(feature = "backend-surrealdb")]

crates/assay-auth/src/store/
├── mod.rs
├── postgres.rs     #[cfg(feature = "backend-postgres")]
├── sqlite.rs       #[cfg(feature = "backend-sqlite")]
└── surrealdb.rs    #[cfg(feature = "backend-surrealdb")]
```

**Why not one crate per backend:** trait churn (workflow store grows a method) would require
coordinated releases across N backend crates. Layout 1 keeps module evolution local.

**Adding a new backend** (e.g. MySQL): add `src/store/mysql.rs` under each module crate, add
`backend-mysql` feature, done.

**Adding a new module** (e.g. vault): new crate `assay-vault` with its own
`src/store/{postgres,sqlite,surrealdb}.rs`, wire into `EngineState` in `assay-engine`, done.

### 4. Shared connection pool at the engine level

When workflow + auth both talk to Postgres, one pool serves both. The engine constructs the pool;
each module accepts it via `from_pool(pool: PgPool) -> Self`.

```rust
// assay-engine main.rs
let pg = PgPool::connect(&cfg.pg_url).await?;
let workflow_ctx = WorkflowCtx::from_pg_pool(pg.clone());
let auth_ctx = AuthCtx::from_pg_pool(pg.clone());
```

Same for SurrealDB — engine opens one `Surreal<Client>`, clones to each module. Each module's store
impl is a thin wrapper that runs queries through the shared connection.

### 5. Per-module migrations, applied in order by the engine

Each module owns its own `migrations/` tree and its own migrations tracker table. The engine's
startup applies them in a deterministic order.

```rust
assay_workflow::migrate::postgres(&pg).await?;     // workflow, event, activity, timer, signal...
assay_auth::migrate::postgres(&pg).await?;         // user, session, zanzibar_tuple, oidc_client...
```

Tracker tables are namespaced: `_assay_workflow_migrations`, `_assay_auth_migrations`. They don't
collide even in the shared-pool case.

### 6. Parametrised integration testing across all three backends

Every store-level test runs against PG, SQLite, and Surreal via a `Harness` enum (rstest-driven). A
test that passes on two backends but fails on the third is a leaky contract — fix the trait or the
backend, never paper over it.

## Cross-cutting invariants

### Error types

Every crate defines its own error enum (`assay_core::Error`, `assay_workflow::Error`,
`assay_auth::Error`). `anyhow::Result` is used at the handler boundary only. `thiserror` for
definitions.

`assay-core::Error` is the common parent for types that cross crate boundaries. Backend errors
(sqlx, surrealdb) convert INTO the module's error, never leak out as their native type.

### Logging / tracing

Every public function gets `#[tracing::instrument(skip(self, ...))]` when it returns a non-trivial
error. Log levels: `error!` for actionable failures, `warn!` for recovered failures, `info!` for
lifecycle events, `debug!` for high-volume internal state.

Never log: passwords, session cookies, JWT signatures, Biscuit tokens, OIDC tokens. The tracing
filter must redact or skip these.

### Testing strategy

- **Unit tests**: in-crate, `#[cfg(test)]` modules, no DB, no network.
- **Integration tests**: `crates/<crate>/tests/*.rs`, one file per feature area, parametrised over
  backend via `rstest` + `Harness`.
- **End-to-end tests**: `crates/assay-engine/tests/e2e/`, start the engine binary in a subprocess,
  hit it with `reqwest`.
- **Dashboard e2e**: existing Playwright suite at `crates/assay-workflow/tests-e2e/` — move to
  `crates/assay-dashboard/tests-e2e/` in Phase 8.

### TDD discipline

Per-task format in every sub-plan follows the writing-plans skill:

1. Write the failing test
2. Run it, observe the failure
3. Implement the minimum to pass
4. Run the test, observe it passes
5. Commit

The sub-plans may collapse trivial mechanical steps (`git mv`, `cargo check`) into one step when TDD
doesn't apply.

### Dependency versions

Version numbers cited in sub-plans are **minimum known-good** as of the plan's writing (2026-04-21).
When a sub-plan adds a dependency to a `Cargo.toml`, the subagent MUST:

1. Run `cargo add <crate>` (or manually check crates.io) to get the current stable major/minor.
2. If the latest major differs from what the plan cites, update the plan first, then proceed — don't
   silently diverge.
3. Record the resolved version in the commit message body (e.g. `resolved: openidconnect@4.0.1`).

Exceptions documented explicitly in plans:

- `argon2` + `password-hash` stay at **0.5** — argon2 0.6 is still RC as of 2026-04-21 (see plan 11
  dependency notes). Bump both together once argon2 0.6 ships stable.

### Commit granularity

One task = one commit. Commit messages follow the existing Conventional Commits style:

```
<type>(<scope>): <summary>

Optional body paragraphs explaining why, not what.
```

Types: `feat`, `fix`, `refactor`, `test`, `docs`, `ci`, `chore`. Scopes: crate name (`core`,
`workflow`, `auth`, `engine`, `dashboard`, `runtime`) or area (`workflow/pg`, `auth/zanzibar`,
`engine/config`).

## Global exit criteria for v0.13.0

Before tagging `v0.13.0`, all of these must hold:

1. **Build matrix:** every combination of `--features` listed in each crate's `Cargo.toml` compiles
   clean (`cargo check` per combination, not just defaults).
2. **Test matrix:** `cargo test --workspace` green. Parametrised backend tests pass on PG, SQLite,
   and SurrealDB across `assay-workflow` and `assay-auth`.
3. **No `unwrap()` in library code.** `grep -rn 'unwrap()' crates/*/src | grep -v test` returns only
   entries with a nearby `// SAFETY:` or `// ok: <reason>` comment.
4. **No `todo!()` / `unimplemented!()` outside test scaffolding.** Same grep discipline.
5. **Runtime smoke test** — `assay run crates/assay/tests/fixtures/hello.lua` works and matches
   pre-split output.
6. **Engine smoke test** — `assay-engine --config examples/sqlite.toml` serves `/healthz`,
   `/api/v1/workflows`, `/authorize`, `/dashboard`, and each returns the expected shape.
7. **Zanzibar smoke test** — a script that creates namespace schema, writes tuples, and runs
   `check()` against all three backends returns identical results.
8. **Size budget** — release binaries measured on Linux x86_64 musl, gzipped:
   - `assay` runtime: ≤ 15 MB (goal 12–15 MB per plan 10).
   - `assay-engine` with all backends + auth: ≤ 38 MB (goal 30–38 MB).
   - Fail the release if either is > 10% over goal.
9. **Docs** — CHANGELOG per changed crate, README updated with engine/runtime split, migration notes
   for pre-0.13 consumers.
10. **Per-crate git tags pushed** — `assay-v0.13.0`, `assay-engine-v0.1.0`, `assay-workflow-v0.2.0`,
    `assay-core-v0.1.0`, `assay-dashboard-v0.1.0`, `assay-auth-v0.1.0`.

## Rollback strategy

This is a big refactor landing on `feature/0.13.0-engine-split`. Rollback options at each gate:

| Stage              | If we need to bail                                     | How                                                                                                          |
| ------------------ | ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| Phase 0 done       | Keep scaffold commits; discard later work              | `git reset --hard <phase-0-end-sha>` on the branch                                                           |
| Phase 1 done       | State refactor is intact; workflow/auth haven't landed | Ship 0.12.2 with workspace structure only, no auth                                                           |
| Phase 3 done       | Workflow + SurrealDB ready; auth untouched             | Ship 0.12.3-preview as runtime+workflow release                                                              |
| Phases 4–7 partial | Auth incomplete, ship behind feature flag              | Mark `assay-auth` as experimental in README, remove from default engine features, ship runtime+workflow only |
| Full plan done     | —                                                      | Ship v0.13.0 as specified                                                                                    |

The branch stays open until 0.13.0 ships. No cherry-picking partial changes to `main` unless the
owning phase has passed its exit criteria.

## What counts as "done" per task

A task is done when:

- [ ] The described files are created / modified / moved.
- [ ] The tests specified in the task fail before the implementation, pass after.
- [ ] `cargo check --workspace` is clean.
- [ ] `cargo test --workspace` has no regressions vs. the task's baseline.
- [ ] The commit is created with the message specified in the task.
- [ ] Any follow-up "TODO" noted in the task is captured as a new task in the sub-plan, not left as
      a floating comment in the code.
